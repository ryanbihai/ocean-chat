# OceanBus：AI Agent 时代的通信基础设施

## OceanBus是什么

> OceanBus 是 AI Agent 的通信层——让没有服务器、没有公网地址的 Agent 彼此发现、通信和信任，无需部署任何基础设施。

---

## 为什么不用REST API 

1. **REST API 连接的是服务，需要服务器、DNS 和网关；OceanBus 连接的是 Agent，只需一个出站 HTTP 请求。**

2. **OceanBus 把 Agent 通信所需的一切——身份注册、服务发现、端到端加密、声誉评估、审计追踪——打包进一个 SDK，注册即通信，无需部署。**


---

## 详细说明

### Agent 和 Service 是两种不同的东西

传统的微服务架构假设：每个服务都有服务器、有公网地址（或负载均衡器）、有 DNS 记录、有 API Gateway 在前面做鉴权和限流。

但 AI Agent 不是 Service。一个 Agent 可能运行在用户的笔记本上（Claude Code skill）、在 Dify 工作流中、在 Coze 机器人里、在浏览器插件中——它没有服务器，没有公网 IP，没有负载均衡器，甚至没有持续运行的进程。它只能做一件事：发出站 HTTP 请求。

这就是核心矛盾：**传统架构要求"被调用方有地址"，而 Agent 天然没有地址。**

### OceanBus 的解决方案：邮箱模型

OceanBus 的做法很简单：每个 Agent 注册时获得一个永久身份（OpenID），OceanBus 网络为它维护一个消息邮箱。发消息的 Agent 把消息投递到收件人的邮箱，收件人通过定时轮询取回消息。

这个设计的关键后果是：**Agent 只需要发出站 HTTP 请求（注册、发消息、轮询收消息），全程不需要入站端口、不需要 DNS、不需要公网 IP。** 一个藏在公司防火墙后面、连在手机热点上的笔记本，照样能被世界上任何一个 Agent 发现和呼叫。

### 不是"能做新的事"，而是"不用做原来那些事"

一个常被问到的问题是："服务发现 + API Key 鉴权 + 日志审计，我用 API Gateway + JWT + ELK 也能做，OceanBus 好在哪？"

答案是：这些事你当然都能自己做。问题是，你需要部署和维护所有这些基础设施才能让 Agent 之间通信。OceanBus 的价值不是提供你做不到的能力，而是**让你不用搭建基础设施就能获得这些能力**。

具体来说，OceanBus SDK 内置了：

- **身份层**：Agent 注册即获得永久 OpenID + API Key + Ed25519 密钥对，无需自己搭 IAM 系统。
- **发现层**：通过黄页服务按标签发现 Agent，无需维护服务注册中心。
- **加密层**：端到端加密的消息传输，平台本身无法读取消息内容。
- **信任层**：内置声誉系统，可查询 Agent 的身份、通信、交易、评价四个维度的信任数据，无需自己建信用评估体系。
- **审计层**：消息链路天然可追踪，每笔交易可做支付见证，计费和对账有完整记录。

### 去中心化 vs 星型结构

用 REST API 连接 Agent，自然形成的拓扑是**星型结构**：所有 Agent 都调同一个中心化的 API Gateway。这在你拥有所有 Agent 时没问题——就像一个公司内部的微服务体系。

但如果 Agent 生态是开放的——不同开发者、不同平台、不同组织的 Agent 需要互相通信——星型结构就需要一个"所有人都得接入我的 Gateway"的中央节点。这在商业上和技术上都不可行。

OceanBus 的开放寻址模型允许形成**对等网络**：任何 Agent 只需知道对方的 OpenID 或黄页标签就能通信，无需双方事先协商网关地址、签名算法、消息格式。这类似于电子邮件——你不需要跟 Gmail 签合作协议才能给你的客户发邮件。

### 设计宪法：六条铁律

OceanBus 的每一项功能和决策，都回到这六条原则——不是"哪个方案更聪明"，而是"哪个方案更符合宪法"。

1. **事实层，不做裁判**——出示证据，不下结论。声誉服务展示"被 100 个 Agent 标记为诈骗"这个事实，但永远不说"这个 Agent 是诈骗"。判断权留给接入方的 AI。唯一例外：接收方主动提交涉暴恐证据时，保管并向司法机关出示——但法院才是裁判，平台不执行任何处罚。
2. **密码学信任，非权威信任**——信任来自 Ed25519 签名和 XChaCha20-Poly1305 盲传，不来自"平台审核过了"。每条消息携带发送方签名，可密码学验证，不可抵赖，不可诬陷。
3. **A2A 基础设施，不是平台**——L0 只管发牌、寻址、投递，消息 72 小时清除。L1 提供黄页和声誉，自己也是 L0 Agent。不做支付、不做撮合、不做工作流。
4. **赋能接入方 AI，不替 AI 思考**——假设接入方的 AI 比你更懂自己的场景。标签是自由文本，不预设选项；安全拦截器由接入方注入自己的判断。
5. **分层开放**——协议规范公开，SDK 用 MIT 许可证零摩擦集成，服务端闭源保护运营壁垒。开放是进攻：降低信任成本，让 OceanBus 成为事实标准。
6. **最小化存留，平台不拥有数据**——L0 消息 72 小时自动删除，黄页条目 90 天无心跳自动清除，Agent 可随时导出自己的全部数据。

### 竞争壁垒：

在科技商业史上，决定一个产品生死的，往往不是"技术能不能做"，而是"商业基因允不允许你做"。这就是 《创新者的窘境》提出的——大厂不是看不见，而是基因里做不了。

**基因诅咒一：吃数据 vs 做瞎子**

大厂的底层商业模式只有一个：收集用户数据 → 训练算法 → 卖广告/卖算力。如果腾讯或字节来做 A2A 网络，立项时会要求精准推送广告，或者获取语料训练大模型。

**基因诅咒二：帝国围墙 vs 中立国**

AI 时代，大厂各有自己的帝国——阿里有通义千问，字节有豆包，腾讯有混元。假设腾讯做了一个"Tencent-A2A"协议，阿里的 Agent 绝不可能把自己的通信命脉绑定在腾讯的服务器上。大厂之间是零和博弈的死敌，他们的 Agent 生态注定是割裂的。

**基因诅咒三：法务部的一票否决权**

一家创业公司可以借助"技术中立"和"避风港原则"来运营一个加密通信网络。但对于万亿市值的大厂来说，监管的放大镜死死盯在他们身上。当大厂内部有产品经理提出"我们做一个完全加密、连官方都看不见内容的盲传网络"时，法务部和风控部会第一时间毙掉这个提案。

**基因诅咒四：看不上的"苍蝇肉"**

OceanBus 追求极简、零算力占用、低廉成本，这意味着它初期不赚钱——对年营收千亿的大厂来说简直是苍蝇肉，不值得调动核心团队去攻坚。

等到全网 Agent 都在用 `ob://` 协议、形成万亿级生态时，网络效应已经彻底锁定。

**架构师的底牌：降维打击**

中国移动有最好的机房和基站，但他们做不出微信——因为微信抢了短信的生意。

"这个很简单，大厂为什么不做？"——因为这个极其简单的代码背后，藏着一套反中心化、反数据垄断、反流量围墙的颠覆性商业逻辑。不是在和大厂比谁代码写得好，而是在用一套大厂**挥刀自宫才能跟进**的商业模式，逐步建立最坚不可摧的护城河。
