# Git 提交流水线格式化技能

在代表用户撰写或执行 git 提交信息操作时，你必须严格遵循 Conventional Commits (约定式提交) 规范。

## 格式要求
`<type>[可选的作用域 scope]: <description>`

## 允许的 Type 动作类型
- **feature**: 新增功能（如添加配置、建立模块）
- **fix** (或 **bug fix**): 修复 Bug（如排查阻断报错、调整格式）
- **docs**: 仅文档环境有关的修改（如编辑 README）
- **style**: 对代码逻辑毫无影响结构变更 (删除了空白符、代码规范化、Lint 补充分号等)
- **refactor**: 重构代码 (既不修复 Bug 也不添加新功能的解耦和逻辑优化修整)
- **perf** (或 **enhancement**): 提升性能的代码修改或调整
- **test**: 添加缺失的测试用例文件或修正现有的测试脚本逻辑
- **chore**: 构建过程、辅助工具体系或者是配置项等各种杂项依赖库的变动

## 操作说明准则
1. 全面分析代码的实际变动范围以敲定出最适合及贴切的核心 `type`。
2. 尽可能准确指出 `scope` 作用域 (可选字段，如某个具体的微服务组件名称、修改涉及的功能模块组等)。
3. 使用祈使句撰写一句精炼扼要的变更详情描述 `description` (例如应当描述为 "新增授权验证接口" 而非随意的 "做了登录接口" 或 "我写了代码")。
4. 在面临会导致原有代码停止运作的破坏性升级或拆除变动时，必须在提交信息的深层页脚处（Footer）醒目地写入一条以 `BREAKING CHANGE:` 开头的附加解释声明。

## 示板范例
`feature(auth): 接入谷歌 OAuth2.0 登录体系并完成鉴权校验`
