# 02-OrderSvc 子应用业务需求说明

## 1. 业务边界与核心职能
处理商城及相关业务场景下的商品订单全生命周期管控。
核心处理域包括但不限于：订单下发（入库）、状态流转、单据溯源查询，以及未来的支付结果回调处理接入（规划中）。

## 2. 场景功能与开放接口逻辑
该系统目前基于 `/api/orders` 的路由根路径对外开发两个基础能力：
1. **`createOrder` (生成订单)**: 提供基础的基于商品 ID 与下单用户的金额扣减下单能力，核定并校验有效负载参数后落下物理表。
2. **`getOrder` (单据详情追溯)**: 依据传入的统一订单流水号（编号），对外提供可靠的单据溯源信息返还能力。

## 3. 标准化与合规性声明
- **API 文档与联调**：当前级目录下附属有一份严格受控的 `openapi.json`，包含了全部前行的 Mock 与请求契约，导入前端管理工具即可生成全量联调报文。
- **数据物理层隔离**：基于微服务严格的数据互斥法则，底层 Mongoose Schema Model 已强制将落库的原始表名重定向至并覆写为专属集合群：`OrderSvc_Order`。避免任意其它可能含有 order 表的服务触发读写幽灵串扰BUG。
