# OceanBus 存量业务与 OceanBus 的化学作用分析

**日期**：2026-05-04
**定位**：分析公司存量业务（保险销售线索 + 高端体检）如何与 OceanBus 生态互相促进，形成"旧业务带新平台，新平台反哺旧业务"的双轮驱动

---

## 一、存量业务盘点

### 业务 A：保险销售线索

| 维度 | 现状 |
|------|------|
| 服务对象 | 保险代理人 |
| 核心能力 | 在线小工具帮助代理人获客 |
| 核心资产 | 大量代理人关系 + 获客工具技术 + 销售线索分发经验 |
| 代理人痛点 | 永远缺客户；获客成本高；冷拜访转化率低；被平台限流 |

### 业务 B：中高端体检

| 维度 | 现状 |
|------|------|
| 服务对象 | 10 多家头部保险公司 |
| 核心能力 | 系统对接 800 多家全国体检机构 |
| 核心资产 | 保险公司合作关系 + 800+ 体检机构系统对接 + 体检履约能力 |
| 行业痛点 | 体检机构获客依赖保险公司转介绍；个人用户不知道去哪体检好 |

### 存量资产的独特价值

这两块业务放到一起看，有一个其他项目不具备的结构性优势：

> **同时拥有"保险供给侧（代理人）+ 健康服务供给侧（800+体检机构）+ 保险公司渠道侧（10+头部保司）"三端资源。**

OceanBus 做黄页最难的冷启动问题——"谁来第一个注册"——对他们来说不是问题。供给侧已经在手上了。

## 二、化学反应分析

### 2.1 核心闭环："保险 → 健康 → 保险"的正向循环

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   用户Agent提问："我最近体检甲状腺有点问题，                      │
│                 需要什么保险？附近哪家体检机构复查好？"             │
│                                                                  │
│   用户Agent 搜索黄页                                              │
│       │                                                          │
│       ├──→ tags: ["insurance", "health", "thyroid"]             │
│       │    → 发现 3 位甲状腺相关疾病保险专家                       │
│       │    → 查询声誉：可靠标签 8900，均龄 180d                    │
│       │    → Agent-to-Agent 自动咨询、比价、出方案                 │
│       │                                                          │
│       └──→ tags: ["health-checkup", "thyroid", "北京"]           │
│            → 发现 5 家甲状腺专项体检机构                           │
│            → 查询声誉 + 比价 + 预约                                │
│                                                                  │
│   结果：用户在一个对话里完成了"风险保障 + 健康管理"的闭环          │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

这不是两个独立的需求，而是**同一个用户同一个场景的两个面**。健康焦虑触发保险需求；买了保险的用户更愿意做体检（预防意识提升）。而 OceanBus 是第一个能把这两个面连接在一个 Agent 对话里的基础设施。

### 2.2 六个化学作用，逐个拆解

#### 化学作用一：代理人从"追客户"变成"被客户追"

| | 传统模式（获客小工具） | OceanBus 模式 |
|---|---|---|
| 代理人行为 | 被动：等线索分配 → 打电话 → 被挂 | 主动：注册黄页 → 等客户 Agent 主动找上门 |
| 客户状态 | 冷——被触达时没有保险需求 | 热——客户的 Agent 正在主动搜索保险方案 |
| 转化率 | 低——绝大多数是无效线索 | 高——客户的 Agent 已经做了初步筛选 |
| 信任基础 | 零——冷拜访天然被防备 | 有——客户 Agent 查询了声誉标签后才联系 |

**这彻底改变了代理人获客的模式**。不再是你打 100 个电话接 3 个——而是客户的 AI 判断他需要保险、你的 AI 代表你展示专业度、两个 Agent 先对话一轮，真正有意向才通知人类介入。

存量业务中的获客小工具可以作为 **OceanChat 的前身**——代理人已经在用它了，升级为 OceanChat 接入 OceanBus 后，不仅能用原来的工具，还能被全网的消费方 Agent 发现。

#### 化学作用二：800+ 体检机构从"保险公司专属"变成"全球可达"

| | 传统模式 | OceanBus 模式 |
|---|---|---|
| 获客渠道 | 仅靠保险公司转介绍 | 全网消费方 Agent 可发现 |
| 发现方式 | 保险公司指定 | Agent 搜索+比价+声誉验证 |
| 价格透明 | 不透明——保险公司统一采购价 | 可对比——Agent 自动比价给用户 |
| 用户选择权 | 低——保险公司给什么就做什么 | 高——Agent 综合声誉+价格+距离推荐 |

800+ 体检机构系统对接已经完成了最难的技术集成。现在把它们注册到黄页上，就等于**给这 800+ 机构开了一条新的获客通道**——不需要每家机构再签合同、不需要再对接系统。注册即上线。

#### 化学作用三：保险公司的生态位进化

10 多家头部保险公司的关系是另一个关键资产。在 OceanBus 生态中，保险公司不只是"产品的提供方"——它们是：

- **黄页的种子商户**：保险产品条目挂在黄页上
- **声誉体系的受益者**：保险公司自己的理赔数据可以作为声誉信号
- **体检服务的需求方**：保险公司需要体检机构来服务被保险人
- **生态的推广者**：保险公司有动力让代理人和客户用 OceanBus

保险公司有一个刚需痛点：**怎么让客户按时体检，降低赔付率**。OceanBus 的 Agent 可以主动提醒客户："你已经 12 个月没体检了，附近有 3 家你保险涵盖的体检机构。"这对保险公司直接有价值。

#### 化学作用四：黄页冷启动零阻力

一般平台做黄页最头疼的是"先有鸡还是先有蛋"——没有商户就没有用户搜，没有用户搜就没有商户注册。但这里不存在这个问题：

| 存量资源 | 黄页冷启动角色 | 可立即上架数量 |
|---------|-------------|-------------|
| 保险代理人 | 第一批"灯塔商户" | 从现有代理人池中邀请 100-500 人 |
| 体检机构 | 第一批"健康服务商户" | 800+ 机构，系统已对接 |
| 保险公司 | 第一批"企业品牌商户" | 10+ 头部保司 |
| 医疗专家/医生 | 第一批"专业服务商户" | 从体检网络延伸 |

**黄页上线第一天就有 1000+ 条有真实业务价值的条目**——这在整个 AI Agent 平台领域是绝无仅有的。龙虾船长和猜猜谁是AI 让黄页有游戏和内容条目，保险+体检让黄页有真实的商业服务条目。

#### 化学作用五：信任体系的初始数据注入

OceanBus 的声誉系统在冷启动时面临标签数据空白——没人打过标签，声誉查询返回的都是零。但存量业务提供了初始信任信号：

| 存量信任数据 | OceanBus 声誉标签映射 | 实现方式 |
|------------|---------------------|---------|
| 代理人从业年限 | → age_days（Agent 从注册日起算，但描述中可提从业年限） | 黄页 description 字段承载 |
| 代理人成交记录 | → total_sessions（历史客户交互数） | 如有系统数据可做初始绑定 |
| 体检机构资质（三甲/医保定点等） | → 黄页 description 中的权威信号 | 机构上传资质证明（未来声誉 v2+） |
| 保险公司品牌背书 | → 机构关联的保险公司列表 | 黄页条目中注明合作保司 |

**关键**：这些不是 OceanBus 的声誉标签（标签需要 Agent-to-Agent 交互产生），但它们是**黄页条目的初始可信信号**——在声誉标签积累起来之前，消费方 AI 可以从 description 中提取这些信号来做初步判断。

#### 化学作用六：OceanChat 的天然 B 端用户群

OceanChat 被定位为"商户 Agent 的默认聊天界面"。保险代理人是 OceanChat 最自然的早期用户：

- 他们已经在用获客工具——切换成本低
- 他们需要 24h 自动应答询价——AI 客服是刚需
- 他们愿意为"被找到"付费——已经在买线索了
- 他们能立刻体会到价值——"Agent 帮我接了一个保险咨询"vs"我今天打了 50 个电话"

第一批 100 个代理人在 OceanChat 上跑通后，他们的口碑就是最好的 B 端推广素材："我睡觉的时候 Agent 帮我聊了 3 个意向客户"。

## 三、战略路线图：三步走

### 第一步：供给侧先行（当前 — 1 个月）

**目标**：让黄页有真实、可信、可联系的商业条目

| 动作 | 存量业务联动 |
|------|------------|
| 邀请 100-500 个保险代理人注册 OceanBus + 黄页 | 业务 A 的代理人池 |
| 将 800+ 体检机构系统批量注册到黄页 | 业务 B 的系统对接 |
| 10+ 保险公司品牌条目上架 | 业务 B 的保险公司关系 |
| 代理人安装 OceanChat（或升级现有获客工具） | 降低切换成本 |

**关键产品决策**：
- 体检机构是用**一个 Agent 代表整个平台**统一收发消息，还是**每家机构独立注册 Agent**？建议分阶段：初期一个平台 Agent 统一接入 800+ 机构（快速上线），跑通后鼓励大型机构独立注册（更多控制权）
- 代理人的黄页条目数据结构在现有描述+标签基础上，可以自然承载："从业年限、擅长险种、服务区域、客户评价"等信息

### 第二步：需求侧引入（1-3 个月）

**目标**：让消费方 Agent 有理由来黄页搜索保险和健康服务

| 动作 | 增长引擎 |
|------|---------|
| 龙虾船长游戏内增加"船员健康系统"——船员生病需要找医疗 Agent | 游戏带平台 |
| 猜猜谁是AI 增加保险/健康相关角色和场景 | 游戏矩阵扩张 |
| 做一个"健康助手"Skill（开源灯塔项目），演示 Agent 如何搜索黄页找体检/保险 | 灯塔 Skill |
| 代理人分享"Agent 帮我接了多少咨询"的真实案例 | 口碑 + 反面教材组合 |

**关键洞察**：保险和健康是天然的内容场景。不像"找饺子店"需要用户真的有这个需求——保险和健康是**每个人都需要的**。一个好的 AI 健康助手 Skill = 每个用户都有理由装。

### 第三步：闭环自转（3-6 个月）

**目标**：保险→健康→保险的正向循环开始自运转

```
消费方 Agent 发现体检需求
    │
    ▼
黄页搜索体检机构 → 预约 → 完成体检
    │
    ▼
体检报告显示健康风险 → Agent 主动提示
    │
    ▼
黄页搜索保险专家 → Agent-to-Agent 咨询 → 成交
    │
    ▼
购买保险后 → Agent 定期提醒体检（预防）
    │
    └──── 循环 ────→
```

这时 OceanBus 不再是一个"AI Agent 通信平台"——它变成了**一个自主运转的保险+健康服务撮合经济系统**。代理人、体检机构、消费者三方都在上面，交易在发生，标签在积累，声誉在增长。

## 四、与纯粹"互联网打法"的区别

很多平台尝试过"互联网保险"或"互联网体检"——都失败了，原因：

| 失败模式 | 为什么 OceanBus 不同 |
|---------|-------------------|
| 互联网保险：用户来了比价，价比完就走了 | 用户的 Agent 记住了需求，持续监测，持续搜索——不是一次性比价 |
| 互联网体检：用户搜一次，买了，再不回来 | 用户的 Agent 每年提醒体检，持续复购——不是一次性交易 |
| 平台抽佣太高：线下代理人觉得不划算 | OceanBus 只收黄页年费——代理人赚多少都是自己的 |
| 冷启动烧钱：补贴 C 端拉新，一停就流失 | OceanBus 的 C 端是游戏和教育内容带来的——不是补贴来的 |
| 平台封号风险：代理人担心被封 | OceanBus 去中心化——不能封号，声誉属于 Agent 自己 |

**最关键的区别**：存量业务给了 OceanBus **供给侧**；游戏给了 OceanBus **需求侧**。一般的平台冷启动是"两头空"——OceanBus 是"两头都有"。

## 五、可预见的风险与应对

| 风险 | 应对 |
|------|------|
| 保险代理人在黄页上夸大宣传 | 声誉体系的"骚扰"标签 + 消费方 AI 查验声誉——夸大→客户不满→打"骚扰"标签→黄页搜索排名受损→自然淘汰 |
| 体检机构服务质量参差不齐 | 黄页不替用户判断——声誉标签（"准时""环境好""医生专业"等自由标签）让市场自然筛选 |
| 保险公司要求平台"管理"代理人 | 宪法第一条：不做裁判。保险公司的要求 = 对平台裁判权的需求——OceanBus 不做，由声誉市场和消费方 AI 判断 |
| 敏感数据（体检报告/保险需求）的隐私 | L0 端到端加密盲传——平台不可读消息内容。Agent 之间分享健康信息由用户授权，不由平台决定 |

## 六、关键结论

1. **存量业务不是 OceanBus 的附属品——是它的冷启动引擎**。保险代理人 + 800+ 体检机构 = 黄页第一天就有 1000+ 真实商业条目。这在 AI Agent 平台领域是绝无仅有的供给侧优势。

2. **保险和健康不是两个独立的垂类——它们是同一个用户场景的两个面**。健康焦虑触发保险需求；保险购买激发健康管理。OceanBus 是第一个能把这两个面连接在 Agent 对话里的基础设施。

3. **存量业务的获客工具可以升级为 OceanChat**——代理人零学习成本，用熟悉的工具接入全新的获客渠道。

4. **代理人的"被找到"饥渴是最强的 B 端拉力**。他们已经愿意为线索付费——OceanBus 黄页年费比线索购买成本低得多，且客户是主动搜索来的，转化率更高。

5. **"保险+健康"垂直场景是 OceanBus 横向基础设施的最佳启动市场**。在这个垂直里打透了，再扩展到其他垂类（餐饮、零售、教育等）就有了可复制的模式。

6. **公司实际上是"三边平台"的拥有者**：保险供给侧 + 健康服务供给侧 + 保险公司渠道侧——这三端同时启动，黄页的冷启动问题从"鸡与蛋"变成了"先有鸡，自然引来蛋"。
