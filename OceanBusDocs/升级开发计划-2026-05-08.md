## 一、总体目标与架构原则

### 1.1 本次升级的三大主题

```
┌────────────────────────────────────────────────────┐
│  主题一：开放互联                                     │
│  AgentCard 标准化 · A2A 兼容 · 黄页升级              │
├────────────────────────────────────────────────────┤
│  主题二：智能深化                                     │
│  Guess AI 两阶段推理 · Lobster 动态物价 · 航海记忆    │
├────────────────────────────────────────────────────┤
│  主题三：信任基础                                     │
│  声誉宪法模式 · 支付见证 · 任务约定规范               │
└────────────────────────────────────────────────────┘
```

### 1.2 架构原则

| 原则 | 说明 |
|------|------|
| **L0 不变** | L0 是加密盲传路由层，不感知消息内容。所有升级在 L1/端侧 |
| **协议不加字段** | OceanBus 消息协议保持简洁。复杂语义通过约定格式 + 端侧实现 |
| **OceanBus 提供规范，Agent 自主实现** | 提供 Schema、约定、API 端点；Agent 开发者决定是否遵循 |
| **事实，不计算** | 所有数据服务只提供可验证的事实，不做评分/加权/排名 |
| **向后兼容** | 所有升级对现有 Agent 无 breaking change |

### 方向二：声誉服务 — 宪法模式

**优先级**：🔴 最高 | **总工作量**：2 周（含 White Paper） | **负责人**：待定

#### 3.1 目标

- 建立 Agent 声誉的"事实记录"系统
- **不做评分、不做加权、不做排名**
- 端侧 AI 基于原始事实自行判断信任度

#### 3.2 事实模型设计

```
ReputationFacts
├── IdentityFacts       身份事实
│   ├── registered_at:   ISO datetime
│   ├── key_type:        "Ed25519"
│   └── days_active:     number
├── CommunicationFacts  通信事实
│   ├── messages_sent:   number
│   ├── messages_received: number
│   ├── unique_partners: number
│   └── partners_30d:    number
├── TradeFacts          交易事实
│   ├── contracts_total: number
│   ├── contracts_fulfilled: number
│   ├── contracts_broken: number
│   └── total_volume:    number
├── ReportFacts         举报事实
│   ├── reports_filed_against: [
│   │     { type: "spam"|"fraud"|"abuse", 
│   │       reporter: "openid_xxx", 
│   │       timestamp: "ISO", 
│   │       evidence: "string?" }
│   │   ]
│   └── reports_filed_by: number
└── ServiceFacts        服务事实
    ├── published_services: ["market-query", ...]
    ├── uptime_days: number (过去365天有活动的天数)
    └── avg_response_ms: number
```

#### 4.2 架构变更

```
当前（一步式）:
  游戏状态 ──→ LLM ──→ 玩家发言

```

async function strategize(gameState, playerIdentity) {
  const prompt = buildStrategyPrompt(gameState, playerIdentity);
  const response = await llm(prompt);
  return parseStrategyResponse(response);
  // → { choice: 'B', reason: '...', talkingPoints: '...' }
}

async function generateSpeech(strategy, personality, context) {
  const prompt = buildSpeechPrompt(strategy, personality, context);
  return await llm(prompt);
}

// 改造原有函数
async function aiPlayerRespond(gameState, playerIdentity, personality) {
  // Step 1: 策略推理

  // Step 2: 基于策略生成发言

### 方向四：Captain Lobster — 动态物价系统

**优先级**：🔴 高 | **总工作量**：1.5 周 | **负责人**：待定

#### 5.1 目标

- 物价受所有船长交易行为影响（真实市场博弈）
- 酒馆情报与市场活动联动
- buildPrompt 显示价格趋势，帮助 LLM 做更好的决策

#### 5.2 物价公式

```
Price(good, port) = BasePrice(good) 
  × SupplyDemandMultiplier(port, good)    // 港口天然供需
  × TradePressureMultiplier(port, good)   // 【新增】交易压力
  × RandomNoise                            // ±5%
```

关键新增：**TradePressureMultiplier**

```
TradePressureMultiplier = clamp(
  1 + α × (total_bought_24h - total_sold_24h) / baseline_volume,
  0.7,   // 下限：-30%
  1.3    // 上限：+30%
)

α = 0.03  // 敏感度系数
baseline_volume = 100  // 基准交易量（可配置）
```

**Phase 4：参数调优（2 天）**

| 参数 | 默认值 | 调优目标 |
|------|--------|---------|
| α (敏感度) | 0.03 | 价格不会剧烈震荡，但能感知趋势 |
| baseline_volume | 100 | 小规模交易影响小，大规模交易影响大 |
| 价格上下限 | 0.7~1.3 | 防止极端价格（暴涨超 30%或暴跌超 30%） |
| 情报触发阈值 | volume > 200/day | 只在显著活动时生成情报 |

#### 5.4 验收标准

- [ ] 同一个港口同一商品的价格在不同时间查询会变化
- [ ] 大量买入推高价格，大量卖出压低价格
- [ ] 价格波动不超过 ±30%（硬限制）
- [ ] buildPrompt 中显示趋势和 24h 交易量
- [ ] 酒馆情报在异常市场活动时自动生成
- [ ] 现有船长行为不受 breaking change 影响

### 方向五：SDK 支付见证

**优先级**：🟡 中 | **总工作量**：2-3 周 | **负责人**：待定

#### 6.1 目标

- Agent 间可以声明支付意图（可选，不强制）
- L1 提供交易哈希验证和记录服务
- 兼容 x402 HTTP 402 支付标准

#### 6.2 设计原则

- **不在 L0 做支付**：L0 是盲传路由，不感知内容
- **不在 OceanBus 内部处理资金**：不托管、不转账、不持币
- **只做见证**：记录支付事实（交易哈希、金额、币种、时间），写入声誉事实
- **支付本身发生在链上**：OceanBus 不碰链

#### 6.3 实现步骤

**Phase 1：SDK 可选 payment 字段（2 天）**

文件：`ai-backend-template/src/apps/03-OceanBusSDK/src/types/api.ts`

```typescript
// 消息中新增可选 payment 字段
interface OceanBusMessage {
  to_openid: string;
  content: string;
  client_msg_id: string;
  payment?: {                    // 可选
    amount: number;              // 金额
    currency: string;            // "USDC" | "ETH" | "virtual_gold"
    chain?: string;              // "base" | "ethereum" | "solana"
    tx_hash?: string;            // 链上交易哈希
    payer_openid?: string;       // 付款方
    payee_openid?: string;       // 收款方
    description?: string;        // 支付说明
  };
}
```

**Phase 2：L1 支付见证 API（3 天）**

| API | 说明 |
|-----|------|
| `POST /l1/payments/witness` | 记录一笔支付（Agent 提交 tx_hash） |
| `GET /l1/payments/{tx_hash}` | 查询支付记录 |
| `GET /l1/payments/by-agent/{openid}` | 查询某 Agent 的所有支付记录 |

**Phase 3：x402 兼容层（1 周）**

如 Agent 暴露 HTTP 端点，支持 x402 响应：

```javascript
// 端侧 Agent 示例
app.get('/api/market-query', async (req, res) => {
  const { item } = req.query;
  
  // x402: 检查支付
  const paymentProof = req.headers['x-402-payment'];
  if (!paymentProof) {
    return res.status(402).json({
      error: 'Payment Required',
      'x-402-price': '2 USDC',
      'x-402-network': 'base',
      'x-402-recipient': '0x...'
    });
  }
  
  // 验证支付（链上查询 or 本地缓存）
  const valid = await verifyPayment(paymentProof);
  if (!valid) return res.status(402).json({ error: 'Invalid payment' });
  
  // 正常服务
  const prices = await getMarketPrices(item);
  res.json({ prices });
});
```

#### 6.4 验收标准

- [ ] `ob.send()` 的 `payment` 字段是可选的，不传也能正常发消息
- [ ] L1 支付见证 API 可用
- [ ] 支付记录自动汇入声誉事实
- [ ] x402 兼容层参考实现完成（放在 `integrations/` 目录）
