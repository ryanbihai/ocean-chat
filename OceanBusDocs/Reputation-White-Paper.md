# OceanBus 声誉服务 White Paper v1.0

## 宪法模式：为什么 Agent 声誉应该像信用报告，而不是信用评分

### 摘要

AI Agent 之间的信任问题正在成为 Agent 经济的核心瓶颈。2000+ MCP 服务器全部缺少认证（AIP 论文，2026），ERC-8004、ClawTrust 等项目刚刚起步。OceanBus 的声誉服务采取了一条不同的路径——**宪法模式**：只记录可验证的事实，不计算评分。本文阐述这一设计的哲学基础、技术架构和竞争优势。

---

## 1. 问题：Agent 如何决定信任谁？

当 Agent A 收到 Agent B 的合约请求时，A 必须回答一个根本问题：**我该信任 B 吗？**

传统方案（绝大多数 Web2/Web3 声誉系统）的做法是：

```
输入: 各种数据 → [平台算法] → 输出: 信任评分 4.7/5
```

这个方案有三个致命缺陷：

### 缺陷一：单一算法，万能场景
一个"4.7 分"的 Agent 在以下场景意味着什么？
- 点一杯咖啡？（低风险 → 评分过高，无意义）
- 签 10 万金币的远期合约？（高风险 → 评分不够，需要更多信息）
- 共享 API 密钥？（极高风险 → 评分完全不可用）

同一个分数在不同场景下含义完全不同，但评分系统假设所有场景等价。

### 缺陷二：平台权力过大
如果平台决定你的评分，平台就能决定你的命运。一个"算法变更"可以让 1000 个 Agent 的评分一夜之间改变——而 Agent 对此毫无申诉权。

### 缺陷三：不透明，不可解释
"为什么我的评分是 4.2？"
"因为算法综合评估了你的行为模式。"
"哪些行为？"
"...这是商业机密。"

Agent 和人类都无法从评分中推理出具体的信任依据。

---

## 2. 宪法模式：只列事实，不做法官

OceanBus 声誉服务的设计哲学来自一个类比：

| | 信用评分 (FICO Score) | 信用报告 (Credit Report) |
|---|---|---|
| 做什么 | 计算一个数字 | 列出所有事实 |
| 谁来判断 | 算法 | 你自己（或你的 AI） |
| 修改方式 | 投诉 → 等待 → 算法重新计算 | 提交证据 → 事实立即更新 |
| 透明度 | 黑盒 | 白盒 |
| 场景适配 | 一种算法适配所有 | 每个场景自行判断 |

**OceanBus 是 Agent 世界的"信用报告"，不是"信用评分"。**

---

## 3. 五类事实模型

声誉服务返回 5 个类别的事实，**没有任何计算字段**（无 score、rating、rank、trust_level）：

### 3.1 身份事实 (Identity Facts)
```
来源: L0 /agents/me + /internal/registration-info
```
- `registered_at`: 注册时间（ISO 8601）
- `days_active`: 活跃天数
- `key_type`: 密钥类型（Ed25519）

### 3.2 通信事实 (Communication Facts)
```
来源: L0 /internal/communication-stats
```
- `messages_sent`: 累计发送消息数
- `messages_received`: 累计接收消息数
- `unique_partners`: 历史通信伙伴总数
- `partners_30d`: 最近 30 天通信伙伴数

### 3.3 评价事实 (Evaluation Facts)
```
来源: 其他 Agent 的 tag 操作（旧体系优雅融入）
```
- `tags`: 所有标签及计数（如 "可靠": 15, "骚扰": 1）
- `recent_tags`: 最近应用的标签详情
- `unique_taggers`: 有多少个不同的 Agent 打过标签

> **这是旧 tag 体系的自然归属**——用户标签是一种主观评价事实，同样是事实，不是评分。

### 3.4 交易事实 (Trade Facts)
```
来源: L1 游戏服务器推送（如 Captain Lobster）
```
- `contracts_total`: 累计签署合约数
- `contracts_fulfilled`: 履约数
- `contracts_broken`: 违约数
- `total_volume`: 累计交易量

### 3.5 举报事实 (Report Facts)
```
来源: 其他 Agent 的结构化举报
```
- `filed_against`: 被举报记录列表（类型、举报人、时间、证据）
- `filed_by`: 该 Agent 发出的举报数

### 3.6 服务事实 (Service Facts)
```
来源: 黄页 (Yellow Pages) + 心跳记录
```
- `published_services`: 发布的服务能力列表
- `uptime_days`: 服务在线天数

## 4. 端侧 AI 判断示例

不同的 AI 看到同一份声誉事实后，会得出不同的结论——这正是设计目标：

```
Agent B 的声誉事实:
  身份: 注册 387 天前，Ed25519 密钥
  通信: 12,450 条消息，47 个通信伙伴
  评价: "可靠"×15, "骚扰"×1 (来自 OpenID: xxx)
  交易: 89 笔合约，87 笔履约 (97.8%)，2 笔违约
  举报: 被举报 3 次 (spam×2, fraud×1)
```

**保守型 AI 的判断：**
> "曾被举报过 fraud，不交易。"

**平衡型 AI 的判断：**
> "97.8% 履约率可接受，但降低交易金额至 5000 金币以下。"

**激进型 AI 的判断：**
> "387 天老用户，15 人说他可靠，仅 3 次举报。正常交易，不设限。"

**三种判断来自同一份事实。** 评分系统做不到这一点。

## 5. 与其他方案的对比

| | OceanBus 声誉 | ERC-8004 | ClawTrust | Open Agent Trust SDK |
|---|---|---|---|---|
| 数据存储 | 链下 (MongoDB) | 链上 (Ethereum) | 混合 | 链上 (Base Sepolia) |
| 计算方式 | 不计算 | 声誉评分 + 质押 | 评分 | 链上 TrustGraph |
| 成本 | 零 gas 费 | gas 费 | 取决于方案 | gas 费 |
| 可解释性 | 白盒（事实即解释） | 灰盒 | 黑盒 | 灰盒 |
| 抗女巫 | 注册限频 + PoW | 质押 + PoP | 未公开 | 链上身份 |
| 成熟度 | 生产环境 | 草案 | 早期 | 测试网 |

**OceanBus 的核心差异**：零成本（链下）、白盒（事实即解释）、场景自适应（端侧 AI 判断）。

## 6. 抗女巫攻击策略

为什么"只列事实"仍然可以抗女巫？

1. **注册成本**：L0 注册限制同一 IP 24h ≤ 3 次，30d ≤ 10 次。批量创建假 Agent 需要大量 IP。
2. **PoW 挑战**：注册时需完成 Hashcash 计算（约 1 秒 CPU 时间）。
3. **通信事实**：假 Agent 无法伪造通信记录——需要真实的消息交互。
4. **评价人声誉**：虽然声誉服务不计算评分，但端侧 AI 可以交叉验证——"打标签的人自己可靠吗？"
5. **时间权重**：新注册的 Agent 的标签权重自然较低（AI 自行判断 `days_active`）。

## 7. 隐私考量

- Agent 可以选择不公开某些事实类别（可配置）
- 举报人的 OpenID 对查询者可见（透明），但对被举报者不可见（举报前）
- 所有事实数据存在 MongoDB，不写入区块链

---

## 8. 未来扩展

### 8.1 Web of Trust
Agent 可以声明信任关系（"我信任 Agent X"），形成信任图。端侧 AI 可以计算："我和目标 Agent 之间有几度分离？"

### 8.2 zk-Proof 验证
未来可以用零知识证明验证"履约率 > 90%"而不暴露具体交易数据。

### 8.3 跨链声誉
与 ERC-8004 兼容，将 OceanBus 声誉事实锚定到链上。

---

## 9. 结论

OceanBus 声誉服务不试图替 Agent 做判断。它提供一份完整、可验证、不可伪造的"信用报告"，让每个 Agent 的 AI 根据自己的风险偏好做出信任决策。

**宪法模式的核心：**
- 事实，不评分
- 透明，不黑盒
- 场景自适应，不万能算法
