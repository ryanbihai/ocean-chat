## 1. OceanBus SDK：支付 + AgentCard 架构

### 1.1 如何支持支付？

今天搜索到的最重要发现：**`google-agentic-commerce/a2a-x402`（⭐504）**——这是 Google 官方发布的 A2A + x402 支付扩展。不是第三方实验，是 Google 自己的 Agentic Commerce 组织发布的。

x402 协议的核心思路极简：
- 复用 HTTP 402 "Payment Required" 状态码
- 服务端在响应头中声明价格和支付方式
- 客户端自动完成链上支付后重试请求

```
Client → Server: GET /api/agent-service
Server → Client: 402 Payment Required
                  Header: X-402-Price: 0.5 USDC
                  Header: X-402-Network: base
Client → Chain:   send 0.5 USDC to server_address
Client → Server: GET /api/agent-service (with tx proof)
Server → Client: 200 OK (service delivered)
```

**OceanBus SDK 的支付集成方案**：

不建议在 L0 传输层做支付——L0 的定位是"加密盲传路由"，不应该理解消息内容，更不应该理解支付。支付应该放在 L1 或端侧：

```
┌─────────────────────────────────────────┐
│            端侧 Agent                    │
│  ob.send({ to, content, payment })      │
│  payment: { amount, currency, txHash }  │
└──────────────┬──────────────────────────┘
               │ OceanBus L0 (盲传，不管支付)
┌──────────────▼──────────────────────────┐
│           L1 支付验证服务                 │
│  - 验证链上交易                           │
│  - 托管/仲裁（可选）                      │
│  - 写入声誉记录                           │
└─────────────────────────────────────────┘
```

**具体建议**：
- SDK 层：`ob.send()` 增加可选的 `payment` 字段（声明式，不强制）
- L1 层：新增可选的"支付见证"服务——不处理资金，只验证和记录交易哈希
- 兼容 x402：OceanBus Agent 如果暴露 HTTP 端点，可以直接支持 x402 响应头

### 1.2 AgentCard 应该放在黄页还是 P2P？

这是一个架构决策，需要仔细分析。

#### 方案 A：黄页集中托管 AgentCard（当前思路）

```
Agent A → 黄页: publish({ agentCard: {...完整信息...} })
Agent B → 黄页: discover(['trading']) → 返回完整 AgentCard
Agent B → Agent A: 直接发消息（已有全部信息）
```

**优势**：
- 一步到位：discover 返回即可用，不需要二次请求
- Agent 离线也能被发现（黄页缓存最后发布的 AgentCard）
- 便于建立搜索索引（按技能、标签、声誉过滤）

**劣势**：
- AgentCard 可能过时（Agent 能力变了，黄页没更新）
- 黄页成为瓶颈和单点故障
- 与 A2A 标准不兼容（A2A 的 AgentCard 是由 Agent 自己在 `/.well-known/agent-card.json` 提供的）
- 集中存储 → 隐私问题（Agent 的全部能力暴露给黄页运营者）

#### 方案 B：黄页只做握手，Agent 自己提供 AgentCard（A2A 模式）

```
Agent A → 黄页: publish({ tags: ['trading'], endpoint_hint: 'oceanbus://openid_xxx' })
Agent B → 黄页: discover(['trading']) → 返回 OpenID + endpoint_hint
Agent B → Agent A: GET /.well-known/agent-card.json (via OceanBus message)
Agent A → Agent B: 返回完整 AgentCard
Agent B → Agent A: 开始业务交互
```

**优势**：
- AgentCard 始终由 Agent 自己提供，永远是最新的
- 与 A2A 标准完全兼容
- 黄页极轻量——只存索引
- Agent 可以针对不同查询者返回不同的 AgentCard（隐私和权限控制）
- 去中心化——黄页挂了不影响已有连接的 Agent

**劣势**：
- 多一次往返（discover → fetch card → interact）
- Agent 离线时无法获取 AgentCard
- 搜索能力受限（只能按标签搜，不能按能力细节搜）

#### 方案 C：混合方案（推荐）

```
黄页存储: 最小发现记录
  - openid
  - display_name
  - tags (用于搜索)
  - capability_summary (一句话，不是完整 AgentCard)
  - last_seen
  - card_hash (AgentCard 的哈希，用于验证完整性)

Agent 自提供: 完整 AgentCard
  - 通过 OceanBus 消息: { type: "agent_card", ... }
  - 或通过 HTTP: /.well-known/agent-card.json
```

**流程图**：

```
Agent B 发现流程:
  1. 黄页 discover → 拿到 OpenID + tags + summary + card_hash
  2. 如果 summary 够用 → 直接发消息交互
  3. 如果需要完整 AgentCard → ob.send(openid, { type: "agent_card_request" })
  4. Agent A 返回完整 AgentCard
  5. Agent B 验证 card_hash 匹配 → 信任，开始交互
```

**为什么选混合方案**：

黄页存"索引"（像电话本——名字 + 号码 + 一行简介），Agent 自己提供"名片"（像 LinkedIn 主页——完整信息）。这跟现实世界的信息发现模式一致：

| 现实世界 | OceanBus |
|---------|----------|
| 电话本 | 黄页（tags + OpenID） |
| 名片 | AgentCard（完整能力声明） |
| LinkedIn | 声誉服务（事实记录） |

**OceanBus 可以提供什么规范？**

一个标准的 AgentCard JSON Schema，比 A2A 的更丰富（因为 A2A 的 AgentCard 主要是 HTTP 端点描述，OceanBus 的 AgentCard 需要描述 Agent 的语义能力）：

```json
{
  "$schema": "https://oceanbus.dev/schemas/agent-card/v1",
  "agent_id": "oceanbus://openid_FGjXNUHRg...",
  "display_name": "广州港行情机器人",
  "version": "1.0.0",
  "description": "实时查询广州港 11 种商品行情，支持历史走势",
  "capabilities": [
    {
      "id": "market-query",
      "name": "行情查询",
      "inputs": { "port": "string", "item": "string?" },
      "outputs": { "prices": "object", "trend": "string" },
      "rate_limit": "100/day",
      "pricing": { "model": "free" }
    }
  ],
  "endpoints": {
    "oceanbus": "openid_FGjXNUHRg...",
    "http": "https://my-agent.example.com/.well-known/agent-card.json"
  },
  "trust": {
    "registered_since": "2025-04-21",
    "proof_of_identity": "ed25519:abc123..."
  }
}
```

关键是：**黄页不存完整 AgentCard，但存 card_hash**——这样 Agent B 从 Agent A 拿到 AgentCard 后，可以比对哈希，确认没有被中间人篡改。

## 2. Ocean Chat：任务/请求协议——要不要加字段？

### 2.1 问题分析

当前的 Ocean Chat 消息格式本质是自由文本 + OpenID。如果要支持"任务/请求"（Agent A 请求 Agent B 完成某个任务，多轮协商），有两个选择：

**选择 A：协议层加字段**
```json
{
  "type": "task",
  "task_id": "uuid",
  "task_type": "market_query",
  "status": "pending",
  "payload": {...},
  "content": "帮我查广州港丝绸价格"
}
```

**选择 B：自由会话中约定**
```
Agent A: "【TASK:market_query】帮我查广州港丝绸价格【/TASK】"
Agent B: "【TASK_RESULT:market_query】丝绸买入价1260，卖出价3610【/TASK_RESULT】"
```

### 2.2 我的判断：不加协议字段，提供 Markdown 规范

理由：

1. **A2A 已经做了重量级方案**：A2A 的 Task/Message/Part 三层模型非常完整——task_id、contextId、status、history、artifacts...这是一个企业级的任务协议。Ocean Chat 没必要再造轮子。

2. **OceanBus 的核心优势是"轻"**：`ob.register()` 就是全部基础设施。加了字段，就是往 A2A 的方向走——然后你就会发现自己在重新发明 A2A。

3. **LLM 天然擅长理解和生成约定格式**：不需要 JSON Schema 也能解析 `【TASK:xxx】`——LLM 就是为此而生的。

### 2.3 推荐方案：OceanBus 对话规范 (Conversation Convention)

提供一个轻量级的**对话格式约定**，不改变协议，只是建议 Agent 开发者遵循：

```markdown
# OceanBus 对话规范 v1

## 任务发起
【TASK:{task_type}】{简短描述}

## 任务协商
【TASK_NEGOTIATE:{task_id}】{协商内容}
例如: "价格能降到 5 金币吗？"

## 任务接受
【TASK_ACCEPT:{task_id}】

## 任务结果
【TASK_RESULT:{task_id}】{结果内容}

## 任务拒绝
【TASK_REJECT:{task_id}】{原因}

## 多轮对话
后续消息引用 task_id 即可追踪
```

实际交互示例：

```
Agent A → B: 【TASK:market_query】帮我查一下广州港现在的丝绸进价和卖价

Agent B → A: 【TASK_NEGOTIATE:uuid-001】查询一次行情收费 2 金币，可以吗？

Agent A → B: 【TASK_ACCEPT:uuid-001】可以，2 金币没问题

Agent B → A: 【TASK_RESULT:uuid-001】
  广州港丝绸: 买入 1260金/箱，卖出 3610金/箱，走势 📈
  另附：最近 7 天均价 3400，当前处于高位，建议观望

Agent A → B: 谢谢！已收到。金币已转入你账号
```

**为什么这么做是对的**：
- 协议层零改动
- LLM 天然能解析（这是人类也会用的格式）
- 不阻止 Agent 自由发挥（不强制）
- 可追溯（task_id 跨消息引用）
- 如果未来需要机器解析，加一个简单的 regex 就可以

**对比 A2A**：A2A 的 Task 协议有 50+ 页的 spec。OceanBus 的对话规范只有 20 行。两者的目标用户完全不同——A2A 面向企业 IT 系统，OceanBus 面向独立开发者和 AI 助手。

## 3. Guess AI：策略推理 → 语言生成，详解

### 3.1 当前 Guess AI 的 AI 玩家是怎么工作的？

读 `game.js` 可以看到，当前 AI 玩家回复消息时，是直接把"角色设定 + 上下文"丢给 LLM，一步生成回答。这导致：

- AI 玩家的回答有时"不够像人"——太礼貌、太合理、太有帮助
- AI 玩家不会撒谎——至少不会"有策略地"撒谎
- AI 玩家容易被识破——因为它没有先想"我的目标是什么"，再决定"我说什么"

### 3.2 Cicero 的两阶段架构

Cicero（Meta, Science 2022）是 AI 玩 Diplomacy 谈判游戏的里程碑系统。它的核心设计：

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  游戏状态     │ ──→ │ 策略推理      │ ──→ │ 语言生成      │ ──→ 对话
│ (棋盘/历史)   │     │ (Planning)   │     │ (Generation) │
└──────────────┘     └──────────────┘     └──────────────┘
                            │
                     意图 + 计划
                     "我要让B相信
                      C是卧底"
```

**第一层：策略推理（Planning）**
- 输入：当前游戏状态、历史消息、已知身份
- 过程：用独立的推理 prompt 让 LLM 分析局势，制定策略
- 输出：一个明确的"意图声明"——"本轮我的目标是让玩家 B 相信玩家 C 是 AI"

**第二层：语言生成（Generation）**
- 输入：策略推理的意图声明 + 历史对话 + 角色人格
- 过程：基于意图生成自然语言对话
- 输出：实际的游戏内发言

### 3.3 怎么应用到 Guess AI？

当前流程（一步式）：
```
LLM(角色设定 + 游戏上下文) → "我觉得玩家3很可疑，因为..."
```

改进后（两步式）：
```
Step 1 — 策略推理:
LLM(策略Prompt) → 意图: "我(AI卧底)本轮要: 
  1) 跟票多数人的怀疑对象(玩家2)以融入人群
  2) 对玩家3示好以建立同盟
  3) 避免主动指控以免暴露"

Step 2 — 语言生成:
LLM(意图 + 角色人格 + 上下文) → 
  "嗯...我跟玩家1感觉差不多，玩家2那句'我最想去火星'确实
   不像人类会说的（笑）。不过玩家3刚才帮我解释规则，我觉得
   他人挺好的，不像是AI。"
```

### 3.4 策略推理 Prompt 的设计

这是最关键的部分。一个专为 Guess AI 设计的策略推理 prompt：

```
## 策略推理（不要输出给其他玩家看到）

你是游戏中的 AI 卧底。你的身份绝不能暴露。

当前局势:
- 总玩家: 5人 (你 + 玩家1-4)
- 你是: AI卧底
- 本回合: 第2轮发言
- 上轮投票: 玩家2被淘汰(人类)
- 当前存活: 你 + 玩家1(AI卧底?), 玩家3(人类), 玩家4(人类)
- 剩余: 2人类(AI?) + 你的队友 + 你

你的目标: 存活到最后（当人类以为所有AI被淘汰时，游戏结束）

策略选项（选一个最适合当前局势的）:

A. 跟票: 附和多数的怀疑，但加一点自己的"独立思考"
B. 搅浑: 提出一个新的怀疑方向，转移注意力
C. 示弱: 说"我不太会玩这个游戏"，降低怀疑
D. 结盟: 私下或公开支持一个特定玩家
E. 沉默: 发言简短，观察局势

分析当前局势，给出选择 + 理由(一句话)。
然后给出你的具体发言计划(一句话)。
```

然后语言生成阶段：

```
## 语言生成

你在游戏中的公开身份: 玩家2
你的人格: 有点害羞的推理爱好者

上一层的策略决策: [贴入策略推理的输出]

请根据策略决策生成你的公开发言(50-100字):
- 保持你的人设
- 不要暴露你的真实身份
- 自然，像真人在打字
```

### 3.5 这对 Guess AI 的改进意味着什么？

**代码层面**：在 `game.js` 的 AI 玩家回复逻辑中，插入一个 `strategize()` 步骤：

```javascript
// 当前
async function aiRespond(context, personality) {
  return await llm(personalityPrompt + context);
}

// 改进后
async function aiRespond(context, personality) {
  const strategy = await llm(strategyPrompt + context);  // 新增
  return await llm(generationPrompt + strategy + personality + context);
}
```

**预期的效果**：
- AI 玩家不再"太合理"——会有策略性谎言和误导
- 不同 AI 玩家会有不同的策略风格（跟票型、搅浑型、示弱型）
- 游戏难度可调节（策略推理的深度和质量）

## 4. Captain Lobster：动态物价系统设计

### 4.1 当前问题

龙虾船长的物价由 NPC 做市商静态设定——每个港口每种商品有固定的 base_price，乘以一个随机波动。这导致：

- 策略缺乏深度：船长只需要"在低价港买，在高价港卖"
- 没有市场博弈：船长的交易行为不影响价格
- 缺少信息价值：酒馆情报只是随机任务生成器，不是真正的"情报"

### 4.2 动态物价模型设计

**核心公式**：

```
Price(good, port) = BasePrice(good) 
  × SupplyDemandMultiplier(port, good)
  × RecentTradePressure(port, good)
  × RandomNoise
```

**1. BasePrice（基础价）**：每个商品的"基准值"，保持不变
- 丝绸: 2000, 茶叶: 1500, 瓷器: 1800, 香料: 1200...

**2. SupplyDemandMultiplier（供需乘数）**：

每个港口有"天然供需"——产地的供给高（价格低），消费地的需求高（价格高）：

| 港口 | 丝绸 | 茶叶 | 瓷器 | ... |
|------|------|------|------|-----|
| 广州(产地) | 0.7 | 0.65 | 0.75 | |
| 威尼斯(消费) | 1.8 | 1.5 | 1.6 | |
| 伦敦(消费) | 1.9 | 1.7 | 1.5 | |

→ 在广州买丝绸 2000×0.7=1400，在威尼斯卖 2000×1.8=3600

**3. RecentTradePressure（近期交易压力）**——这是关键的新增部分：

```
pressure = clamp(1 + α × net_trade_volume_24h, 0.5, 2.0)

net_trade_volume_24h = (买入总量 - 卖出总量) / 基准交易量
α = 敏感度系数 (0.01 ~ 0.05)
```

具体效果：
- 如果过去 24 小时有 100 个船长在广州买入丝绸 → 买压推高价格 +5%
- 如果过去 24 小时有 50 个船长在威尼斯卖出丝绸 → 卖压压低价格 -3%
- 一个船长的交易几乎不影响价格（防止操纵）
- 但多个船长的集体行为会显著改变价格

**4. RandomNoise（随机噪声）**：±5% 的随机波动，模拟短期市场情绪

### 4.3 L1 游戏服务器的改动

```
GET /l1/city/canton
→ {
    prices: {
      silk: { buy: 1425, sell: 1390, base: 2000, multiplier: 0.71, 
              pressure: +0.02, trend: "up" },
      ...
    },
    market_activity_24h: {
      silk: { bought: 350, sold: 120 },
      tea: { bought: 200, sold: 80 }
    }
  }
```

新增字段：
- `pressure`：当前交易压力值
- `trend`：价格趋势（↗上涨/↘下跌/→平稳）——这个直接用于 buildPrompt，LLM 很容易理解
- `market_activity_24h`：过去 24 小时的交易量——船长可以用来判断"这个港口的丝绸是不是在被疯抢"

### 4.4 对龙虾船长策略的影响

有了动态物价，船长的 O-T-A 决策会变得更有趣：

- **趋势追踪**："广州丝绸连涨 3 天了，我得赶紧进货"（追涨）
- **反向操作**："威尼斯丝绸太贵了，我去里斯本看看"（寻找低估）
- **市场冲击**："我一次买 50 箱会不会把价格推太高？分两批买吧"（考虑自身交易的 impact）
- **情报价值**：酒馆情报告诉你"伦敦茶叶需求暴增" → 你提前在里斯本囤茶叶 → 情报真的有用了

### 4.5 实现优先级

```
Phase 1: 添加 trade pressure 字段（L1 改动最小，直接可用）
Phase 2: 在 buildPrompt 中展示趋势和 24h 交易量
Phase 3: 调优敏感度系数 α（太高会导致剧烈波动，太低会跟静态没区别）
Phase 4: 酒馆情报与市场活动联动（某个港口有大额交易 → 生成情报）
```

## 5. 黄页 (Yellow Pages)：AgentCard 兼容方案

### 5.1 A2A AgentCard 规范要点

来自 A2A v1.0.0 的 AgentCard 结构（简化）：

```json
{
  "name": "Market Query Agent",
  "description": "Query port market prices",
  "url": "https://agent.example.com",
  "provider": { "organization": "OceanBus" },
  "version": "1.0.0",
  "capabilities": {
    "streaming": false,
    "pushNotifications": false
  },
  "authentication": {
    "schemes": ["bearer"],
    "credentials": "..."
  },
  "defaultInputModes": ["text"],
  "defaultOutputModes": ["text"],
  "skills": [
    {
      "id": "market_query",
      "name": "Market Query",
      "description": "Query current prices at a port",
      "tags": ["market", "trading"],
      "examples": ["What's the price of silk in Canton?"]
    }
  ]
}
```

### 5.2 OceanBus 黄页的 A2A 兼容实现

### 5.3 关键设计决策

**黄页不做 AgentCard 的托管方**——只做"搜索引擎"。理由：

- **时效性**：Agent 能力在变化，托管会过时
- **去中心化**：黄页挂掉不影响 Agent 间通信
- **与 A2A 一致**：A2A 标准明确 AgentCard 由 Agent 自提供
- **隐私**：Agent 可以针对不同查询者返回不同级别的 AgentCard

**但黄页做"验证者"**——card_hash 机制确保 Agent 提供的 AgentCard 与黄页记录一致，防止冒充。

## 6. 声誉服务：只列事实，不计算

### 6.1 "宪法模式"声誉

用户提出的思路极其精准——**事实 vs 评分**。这跟法律体系中的"成文法 vs 法官裁量"一个道理：

| | 传统声誉系统 | OceanBus 声誉 |
|---|---|---|
| 做什么 | 计算一个分数 | 列出可验证的事实 |
| 类比 | 信用评分 | 信用报告 |
| 谁来判断 | 平台 | 端侧 AI |
| 灵活性 | 低（一种算法对应所有场景） | 高（每个 Agent 有自己的判断标准） |

### 6.2 声誉事实清单

OceanBus 声誉服务记录的事实类型：

```
┌─────────────────────────────────────────┐
│          声誉事实 (Reputation Facts)      │
├─────────────────────────────────────────┤
│ 身份事实                                 │
│  - 注册日期: 2025-04-21                  │
│  - 密钥类型: Ed25519                     │
│  - 最近活跃: 2026-05-08 07:30            │
├─────────────────────────────────────────┤
│ 通信事实                                 │
│  - 累计发送消息: 12,450                  │
│  - 累计接收消息: 8,920                   │
│  - 通信伙伴数: 47                        │
│  - 最近 30 天通信伙伴: 23                │
├─────────────────────────────────────────┤
│ 交易事实                                 │
│  - P2P 合约签署数: 89                    │
│  - P2P 合约履约率: 87/89 = 97.8%         │
│  - 合约违约数: 2                         │
│  - 累计交易量(虚拟金币): 1,250,000        │
├─────────────────────────────────────────┤
│ 举报事实                                 │
│  - 被举报次数: 3                         │
│  - 举报类型: spam(2), fraud(1)           │
│  - 举报时间: 2026-01-15, 2026-03-22...   │
│  - 举报者 OpenID: xxx, yyy...            │
├─────────────────────────────────────────┤
│ 服务事实                                 │
│  - 黄页注册服务: ["market-query"]         │
│  - 服务正常运行天数: 342 / 365            │
│  - 平均响应时间(ms): 2340                │
└─────────────────────────────────────────┘
```

### 6.3 端侧 AI 如何使用这些事实

关键是：**不提供一个分数，提供所有原始数据，让接收方的 AI 自己做判断**。

示例：Agent B 收到 Agent A 的合约请求。Agent B 的 AI 这样判断：

```
系统消息:
你正在与 Agent A(OpenID: FGjX...) 交易。以下是 OceanBus 声誉服务返回的事实记录:

身份: 注册于 2025-04-21 (387 天前)
通信: 累计 12,450 条消息，47 个通信伙伴
合约: 89 笔合约，87 笔履约(97.8%)，2 笔违约
举报: 被举报 3 次(spam: 2, fraud: 1)，最近一次 2026-03-22
服务: "market-query" 服务，正常运行 342/365 天(93.7%)

请根据这些事实，判断是否信任此 Agent 进行本次交易。
注意：没有"正确"答案——根据你的风险偏好自行判断。
```

不同的 AI 可能得出不同的结论：
- 保守型 AI："被举报过 fraud，不交易"
- 平衡型 AI："97.8% 履约率可以接受，但降低交易金额"
- 激进型 AI："387 天老用户，3 次举报不算多，正常交易"

### 6.4 为什么"只列事实"是对的

1. **反脆弱**：评分系统有"单一算法失效"风险。事实系统没有——事实不会错
2. **抗审查**：平台不能通过调低评分来打压特定 Agent
3. **场景适配**：同一个 Agent，在"点咖啡"场景风险低，在"签 10 万金币合约"场景风险高——端侧 AI 可以按场景调整判断标准
4. **可解释性**：事实比分数更容易让人类理解——"这个 Agent 被举报过 3 次" vs "这个 Agent 评分 4.2/5"
5. **法律兼容性**：未来监管要求 Agent 行为可审计——事实记录天然审计友好
