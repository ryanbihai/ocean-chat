## 一、今日要闻概览

1. **Google A2A v1.0.0 正式发布**（2026-03-12，Linux Foundation 托管）——Agent-to-Agent 通信协议进入标准化阶段，SDK 覆盖 Python/Go/JS/Java/.NET

## 三、Agent 身份与声誉——今天的重头戏

> 第三个搜索 Agent 深入挖掘了 Agent 身份、声誉和信任基础设施的最新进展。这是今天最重要的发现板块。

### 5.1 ERC-8004 "Trustless Agents"（以太坊标准草案）
- **发起方**：MetaMask、Ethereum Foundation、Google、Coinbase（2025 年 8 月）
- **核心**：区块链上的 Agent 发现、选择和交互——**无需预先信任**
- **信任模型**：声誉评分（客户反馈）+ 质押担保重新执行 + zkML 证明 + TEE 预言机
- **对 OceanBus 的启发**：
  - **声誉服务**：ERC-8004 的信任模型设计可以直接参考——它是目前最权威的 Agent 信任标准方案
  - **建议行动**：对比 ERC-8004 的信任模型和 OceanBus 声誉服务的设计，写一份对齐分析

### 5.2 ClawTrust（直接竞品）
- **GitHub**：`clawtrust-hub/clawtrust`
- **定位**："Agent 经济的信任基础设施——Agent 在交互前验证、评分、建立信任"
- **为什么重要**：名字和定位与 OceanBus 声誉服务几乎一样！但好在它也是早期项目
- **对 OceanBus 的启发**：
  - **声誉服务**：需要加速。ClawTrust 的 README 已经上 GitHub，虽然可能还没什么用户，但名字占住了
  - **建议行动**：立即注册/保护相关名称（clawtrust 的 npm 包名、域名等如果还没被占用的话），OceanBus 声誉服务的命名和定位要清晰区分

### 5.3 Open Agent Trust SDK（技术参考）
- **npm**：`@open-agent-economy/trust-sdk`
- **GitHub**：`Open-Agent-Economy/open-agent-trust`
- **核心**：TypeScript SDK，链上信任与声誉。智能合约包括 InteractionRegistry、AttestationSchemaRegistry、TrustGraph。部署在 Base Sepolia 测试网
- **对 OceanBus 的启发**：
  - **声誉服务**：这是最接近 OceanBus 需要的技术方案——有代码、有合约、可参考。但它是链上的（gas 费 + 速度慢），OceanBus 的 L0+L1 链下架构有天然优势
  - **建议行动**：深入研究 Open Agent Trust SDK 的代码架构，特别是 TrustGraph 的设计，评估是否可以在 OceanBus L1 上实现一个链下版本

### 5.4 Agent Trust Network（Credit Score for Agents）
- **网站**：agenttrust.network（有反爬保护）
- **核心**：AI Agent 的 "FICO 信用评分"
- **对 OceanBus 的启发**：
  - **声誉服务**：FICO 模型是成熟的信用评估框架——支付历史、欠款额、信用历史长度、新信用、信用类型。Agent 声誉也可以类似：**交易履约率、活跃天数、被举报次数、合约履行率、社区评价**
  - **建议行动**：参考 FICO 模型的 5 个维度，设计 OceanBus Agent 声誉评分的 5 维度模型

### 5.5 AIP — Agent Identity Protocol（arXiv 2603.24775，2026 年 3 月）
- **核心贡献**：提出 IBCT（Invocation-Bound Capability Tokens）——融合身份、授权和溯源的单次调用令牌
- **关键发现**：扫描约 2000 个 MCP 服务器，**全部缺少认证**
- **对 OceanBus 的启发**：
  - **SDK / OpenID**：2000 个 MCP 服务器都没有认证——这正是 OceanBus OpenID 要解决的问题！OceanBus 的密码学票据（XChaCha20-Poly1305 加密 UUID，O(1) 解密）在架构上比 IBCT 更简洁
  - **建议行动**：以 AIP 论文的发现为切入点，写文章论证"Agent 通信必须内建身份"——OceanBus 的设计哲学

### 5.6 AP2 — Agent Payments Protocol（Google）
- **GitHub**：`google/agent-payments-protocol`
- **核心**：基于稳定币的 Agent 间支付协议
- **对 OceanBus 的启发**：
  - **Captain Lobster / 声誉服务**：AP2 解决"Agent 怎么付钱"，OceanBus 解决"Agent 怎么通信"+"Agent 信谁"——三个协议天然互补
  - **建议行动**：长期关注 AP2，未来 OceanBus 的 P2P 合约可以集成 AP2 实现真实支付

### 5.7 x402（HTTP 402 "Payment Required" 复兴）
- **网站**：x402.org
- **核心**：一行代码在服务端集成，让 Agent 为每次 API 调用付费
- **对 OceanBus 的启发**：
  - **黄页**：如果黄页中的 Agent 可以提供付费 API，x402 是天然的计费层
  - **建议行动**：评估 OceanBus SDK + x402 的集成方案——让 Agent 注册时可以声明"我的服务收费 X 美元/次"

### 5.8 NIST AI Agent 标准倡议
- **要点**：NIST 正式启动 Agent 治理、身份和认证框架的行业意见征集
- **对 OceanBus 的影响**：当 NIST 发布 Agent 身份标准时，OceanBus 的 OpenID 应该兼容或对齐——这是成为"合规"基础设施的机会

## 八、OceanBus 产品线深度反思

### 8.1 OceanBus SDK (npm: oceanbus)

**现状**：JS/TS SDK，提供 Agent 注册、消息收发、OpenID 身份、Ed25519 签名。HTTP/2 轮询（2s interval）。L0 盲传路由 + L1 黄页/声誉。

**外部动态映射**：
- A2A v1.0 成熟 → OceanBus 可以作为 A2A 的"轻量替代"
- ACP(⭐1002) 竞争 → 需要突出差异化
- x402 支付 → Agent 微支付是自然延伸

**可学习**：
1. A2A 的 AgentCard 机制（`/.well-known/agent-card.json`）——OceanBus 的 Agent 元数据发布可以标准化
2. ACP 的 API 简洁性——对比后看 OceanBus API 是否可以更简洁
3. WebSocket 传输（ARP 项目）——实时性需求越来越强

**建议行动**（按优先级）：
- 🔴 **高优先级**：实现 OceanBus ↔ A2A bridge（AgentCard 兼容层）——让 OceanBus Agent 能被 A2A 生态发现
- 🟡 **中优先级**：评估 WebSocket 传输通道——对 Guess AI 和 Captain Lobster 的实时性至关重要
- 🟢 **低优先级**：评估 Python SDK——MCP/LangChain 生态需求
- 🟢 **低优先级**：API 简洁性对比（vs ACP），做一轮 DX 优化

### 8.2 Ocean Chat

**现状**：P2P 加密消息，好友系统，OpenID 身份，黄页发布/发现。基于命令行，面向开发者和 AI 助手。

**外部动态映射**：
- A2A 的 Agent 间消息是结构化任务（Task/Message/Part），Ocean Chat 是自由文本——两个不同的抽象层
- WebSocket 实时通信开始流行

**可学习**：

**建议行动**：
- 🟡 **中优先级**：在 Ocean Chat 中引入"任务/请求"概念（不仅是聊天，还可以是"帮我查一下 XX 港的行情"）
- 🟢 **低优先级**：WebSocket 支持（实时聊天体验）
- 🟢 **低优先级**：简化 first-run 体验

### 8.3 Guess AI

**现状**：多人社交推理游戏，1 位裁判 + 4-6 位玩家，P2P 投票，AI 卧底。基于命令行，每轮需要手动执行 node 命令。

**外部动态映射**：
- 社交推理游戏是 LLM Agent 研究的热门领域——12+ 篇论文（2023-2025）
- AvalonBench / AMONGAGENTS / Cicero 提供了成熟的方法论
- Project Sid 展示了大批量 Agent 自主协调的可能性

**可学习**：
1. **AvalonBench 评估框架**：Guess AI 可以引入标准化的 AI 玩家评估——AI 的"人类相似度"和"欺骗成功率"
2. **Cicero 的两阶段架构**（策略推理 + 语言生成）：Guess AI 的 AI 玩家可以升级——先做策略推理（"这轮我应该指控谁？为什么？"），再生成对话
3. **Smallville 的人格建模**：每种 AI 角色有独特的记忆、性格、说话风格——不只是随机选一个答案
4. **多 Agent 编排框架的游戏流程自动化**——裁判角色可以 AI 化

**建议行动**：
- 🔴 **高优先级**：让裁判角色 AI 化——AI 自动分配身份、主持流程、宣布结果。减少玩家执行命令的摩擦
- 🔴 **高优先级**：研究 Cicero 的两阶段架构，升级 Guess AI 的 AI 玩家策略（策略推理 → 语言生成）
- 🟡 **中优先级**：引入 Smallville 式的人格建模——给 AI 玩家不同的"性格参数"，增加游戏可玩性
- 🟡 **中优先级**：增加 Web UI 或 CLI 体验优化（颜色、动画、倒计时）
- 🟢 **低优先级**：持久化游戏状态

### 8.4 Captain Lobster (龙虾船长)

**现状**：Zero-player AI 交易游戏，ReAct 循环，11 种商品 × 10 个港口，P2P 合约，酒馆情报。每 30 分钟自动运行一轮。早晚各一份日报。

**外部动态映射**：
- **Project Sid**（1000 Agent 文明模拟）→ 龙虾船长的最终形态应该是"多船长贸易帝国"而非单人游戏
- **LLM Economist**（Agent 经济模拟）→ 游戏内的物价系统可以更真实——受所有船长交易影响，而非静态 NPC 价格
- **Generative Agents / Smallville**（记忆流架构）→ 龙虾船长需要持久化的"航海经验记忆"
- Ralph Loop(⭐133) 的记忆架构 → 具体可参考的工程实现
- x402 / AP2 支付协议 → P2P 合约可以从"游戏金币"升级到"真实价值"（远期）

**可学习**：
1. **Project Sid 的多 Agent 社会涌现**：龙虾船长的隐藏潜力不是"单人游戏"，而是"多船长经济沙盒"——如果 50 个船长同时运行，会自然形成贸易联盟、价格同盟、垄断竞争。这是任何单机游戏都无法提供的体验
2. **Smallville 的记忆流（Memory Stream）**：分为三层——
   - 观察（Observation）：发生了什么
   - 反思（Reflection）：对观察的高级抽象
   - 计划（Plan）：基于反思的未来行动
   - 龙虾船长的 O-T-A 循环只做了第一步，缺少反思和计划
3. **LLM Economist 的机制设计**：物品价格可以由全局供需动态决定，而非预设的 NPC 买卖价
4. **Ralph Loop 的 4 层记忆栈**：具体工程参考——短期/中期/长期/元记忆

**建议行动**：
- 🔴 **最高优先级**：在 react-engine 的 buildPrompt 中加入"近期策略回顾"（最近 5 轮决策的简要总结 + 盈亏）——让 LLM 有"记忆"，避免重复犯错。这是最小改动、最大收益的优化
- 🔴 **高优先级**：设计"多船长模式"——允许用户激活 2-3 个船长，每个有不同策略。这是从 Project Sid 获得的核心灵感
- 🔴 **高优先级**：参考 LLM Economist，将酒馆情报系统升级为"市场传闻"——由其他船长的聚合交易行为生成（不需要 LLM 成本），增加策略深度
- 🟡 **中优先级**：研究 Smallville 的记忆流架构，设计龙虾船长专属的"航海经验系统"（Observation → Reflection → Plan）
- 🟡 **中优先级**：引入策略基因——每个船长有可学习的交易参数（风险偏好、持仓周期、偏好的商品类型）
- 🟢 **低优先级**：船队协调模式

## 十、执行摘要

今天的搜索揭示了一个快速演化的格局，四个关键发现：

### Agent 通信层：标准已定，差异化在"轻量"
A2A v1.0.0 (Linux Foundation) 和 ACP (⭐1002) 代表了 Agent 通信协议的两条路线——前者企业级重型，后者社区级轻量。OceanBus 的位置是**比 ACP 更完整（自带身份、黄页、声誉），比 A2A 更轻量（零部署）**。战略：做 A2A 兼容层（AgentCard），保持差异化。

### Agent 身份层：2000 个 MCP 服务器全无认证——OceanBus OpenID 的巨大机会
AIP 论文的发现验证了 OceanBus 的核心假设：Agent 世界迫切需要内建的身份系统。OceanBus 的 OpenID（XChaCha20-Poly1305 加密票据，O(1) 解密）已经解决了这个问题——但需要更好地讲述这个故事。

### Agent 信任层：蓝海中的早期竞争者
ClawTrust、Open Agent Trust SDK、ERC-8004 都在做 Agent 信任，但都还处于早期（测试网/草案阶段）。OceanBus 的 L1 声誉模块是目前唯一有生产级架构的方案。**窗口期约 6-12 个月**。本周最重要的行动：写声誉服务 White Paper。

### Agent 社会层：Project Sid 证明了多 Agent 涌现的巨大潜力
1000 个 Agent 自发形成政府和宗教——这验证了 OceanBus 的核心愿景。龙虾船长的终极形态不是单人游戏，而是"多船长贸易帝国"。Guess AI 可以从 Cicero 和 AvalonBench 的学术研究中直接借鉴。

**本周三个必须做的事：**
1. 声誉服务 White Paper（定义 OceanBus Agent 信任模型）
2. 龙虾船长 buildPrompt 加入"近期策略回顾"（5 轮窗口，最小改动最大收益）
3. OceanBus ↔ A2A bridge 技术方案（AgentCard 兼容层）
