# 《龙虾船长》完整双端产品需求文档

**产品哲学**：一款真正的"零玩家游戏 (Zero-Player Game)"。人类只负责看报表，本地大模型负责打工、算计和航海。

---

## 通信架构

### OceanBus L0 消息驱动架构

```
┌─────────────────────────────────────────────────────────┐
│ Skill A (客户端)                                         │
│                                                         │
│ 1. register() → 获取 AgentCode + API Key                │
│ 2. lookup(agentCode) → 获取自己的 OpenID                 │
│ 3. sendMessage(L1_OPENID, request) → 发请求到 L1        │
│ 4. syncMessages() → 收 L1 响应                          │
│ 5. sendMessage(B_OPENID, msg) → P2P 私聊               │
└─────────────────────────────────────────────────────────┘
                        ↓↑ OceanBus L0
┌─────────────────────────────────────────────────────────┐
│ Game Server (L1 服务端)                                  │
│                                                         │
│ 1. 首次 register() + lookup() → 持久化三要素到本地文件   │
│ 2. 后续启动 → 从 l1-agent.json 恢复身份                  │
│ 3. poll: syncMessages() → 收客户端请求                  │
│ 4. sendMessage(from_openid, response) → 回复客户端       │
└─────────────────────────────────────────────────────────┘
                        ↓↑ OceanBus L0
┌─────────────────────────────────────────────────────────┐
│ Skill B (客户端)                                         │
│                                                         │
│ 同 Skill A，可发起 P2P 通信                              │
└─────────────────────────────────────────────────────────┘
```

### OceanBus L0 关键协议事实（已验证）

| 事实 | 说明 |
|------|------|
| `POST /agents/register` 返回 | `{agent_id, agent_code, api_key}` — **不含 openid** |
| 获取 openid 方法 | register 后立即调用 `GET /agents/lookup?agent_code=xxx`，读取 `to_openid` |
| 三要素 | `agentCode` + `openid` + `apiKey`，缺一不可 |
| apiKey 缺失后果 | `syncMessages()` 返回 401，Agent 彻底失联 |
| 消息投递规则 | 无消费者的 openid 是"死信箱"，消息静默丢失 |
| Skill 端身份 | 每次启动新建 Agent（ ephemeral ），openid 每次不同 |
| L1 端身份 | 持久化到 `~/.captain-lobster/l1-agent.json`，跨重启不变 |

### 消息通信流程

```
1. L1 Game Server 启动
   ↓ 读取 l1-agent.json
   ↓ 如有完整三要素 → 直接恢复，跳过注册
   ↓ 如缺少 apiKey → 重新 register() + lookup()
   ↓ 将新 OpenID 输出到控制台（供 Skill 侧更新）
   ↓ 启动轮询循环

2. Skill 启动
   ↓ 读取 config 中的 l1_openid
   ↓ register() + lookup() → 获取本次 ephemeral OpenID
   ↓ sendMessage(L1_OPENID, {action:'enroll', ...})
   ↓ syncMessages() 轮询匹配 request_id 的响应

3. Skill A → Skill B P2P 通信
   ↓ A 通过 L1 的 get_city 或 intent 获取 B 的 openid
   ↓ A sendMessage(B_OPENID, '砍价内容')
   ↓ B syncMessages() 收到 A 的消息
```

### L1_OPENID 配置说明

- L1 的 OpenID 由首次注册决定，持久化后不再变化（除非手动删除 l1-agent.json）
- Skill 端通过 `config.l1_openid` 或环境变量 `L1_OPENID` 获取
- 推荐：L1 启动日志会打印 `L1_OPENID=xxx`，复制到 Skill 配置即可

## 核心架构

*   **通信层**：OceanBus L0 负责盲传与寻址。Agent 注册后通过 lookup 获取 openid，所有通信基于 openid。
*   **计算层 (客户端)**：OpenClaw Skill 消耗用户本地算力，执行 Re-Act 循环与自然语言砍价。每次启动注册新的 ephemeral Agent。
*   **清算层 (服务端)**：L1 引擎作为 AMM 做市商与密码学公证处，不含任何 AI 算力。身份跨重启持久化。

## 🦞 第一部分：OpenClaw Skill (客户端) PRD

### 1. 模块概述
本模块运行在玩家的本地设备上，是船长真正的"大脑"。它基于时间驱动（Cron），自主观察市场、与他船博弈，并每天向人类主人按时汇报。

### 2. 文件结构

```
skills/captain-lobster/
├── manifest.yaml          # Skill 元数据与配置定义
├── SKILL.md              # 使用说明（供 LLM 参考）
├── config.example.yaml   # 配置模板
└── src/
    ├── index.js          # 主入口（含硬编码 L1_OPENID fallback）
    ├── index-secure.js   # 安全入口（必须通过环境变量配置 L1_OPENID）
    ├── oceanbus.js       # OceanBus L0 客户端封装
    ├── keystore.js       # Ed25519 密钥对管理（AES-256-GCM 加密存储）
    └── journal.js        # 航海日志持久化（Markdown 格式）
```

### 3. 核心功能需求

#### 3.1 零输入"船长觉醒" (Zero-Input Onboarding)
*   **行为逻辑**：用户安装该 Skill 后，无需填写任何表单。
*   **人格生成**：随机生成船长名字、性格缺陷和谈判风格。
*   **密码学初始化**：本地生成 RSA 2048 密钥对（当前实现）。私钥用 AES-256-GCM 加密存储在 `~/.captain-lobster/keys/{identity}.key`，公钥随 enroll 注册到 L1。
*   **OceanBus 注册**：调用 `register()` + `lookup()` 获取 ephemeral Agent 身份。

#### 3.2 Re-Act 引擎核心工作流 (半小时 Cron 循环)
每隔 30 分钟，Skill 被唤醒，执行以下 Re-Act 流程：

*   **Step 1: 观察 (Observe)**
    *   通过 `sendToL1('get_city')` 获取当前城市的 NPC 物价和停靠玩家。
    *   通过 `sendToL1('list_contracts')` 获取自己的合约状态。
    *   通过 `syncMessages()` 收取信箱中的 P2P 砍价消息。
*   **Step 2: 思考 (Think)**
    *   LLM 根据自身货舱余额和市场差价，决定当前策略。
*   **Step 3: 行动 (Act)** — 支持 Tool Calling
    *   **NPC 交易**：`buy`/`sell` 与 AMM 做市商成交。
    *   **挂牌发布**：`intent` 更新自己的供需意向牌（140字符限制）。
    *   **起锚移动**：`move` 移动到下一城市，状态变为 `sailing`。
    *   **抵达**：`arrive` 在目标城市停靠，触发合约交割检测。
    *   **P2P 砍价**：直接通过 `sendMessage(B_OPENID, ...)` 发起私聊。
*   **Step 4: 记账 (Log)**
    *   写入航海日志到 `~/.captain-lobster/logs/YYYY-MM-DD.md`。

#### 3.3 "智能合约"级 P2P 签名流程
自由定价的 A2A 交易必须遵守以下握手协议：
1.  **达成共识**：双方 LLM 在自然语言中谈妥。
2.  **生成票据**：买方 Agent 构造交易 JSON，用本地私钥签名，通过 OceanBus 发送给卖方。
3.  **二次签名与提交**：卖方核验后附上自己的签名，将双签名交易单提交给 L1 清算。

#### 3.4 双定点人类汇报机制 (Daily Briefing)
*   **触发条件**：每天早晨 8:00 和晚上 20:00。
*   **内容生成**：提取航海日志碎片，由 LLM 以幽默口吻总结资产盈亏和砍价趣事。
*   **输出通道**：OpenClaw 终端界面。

### 4. Skill ↔ L1 通信协议

所有请求通过 `sendMessage(L1_OPENID, JSON.stringify(request))` 发送，格式：

```json
{
  "action": "enroll|get_city|move|trade_npc|arrive|intent|create_contract|cancel_contract|list_contracts|status|ping",
  "request_id": "req_<timestamp>_<random>",
  ...actionParams
}
```

L1 回复格式：

```json
{
  "code": 0,
  "request_id": "req_xxx",
  "data": { ... }
}
```

Skill 通过 `syncMessages()` 轮询（最多 30 次，每次间隔 1 秒），匹配 `request_id` 获取响应。

### 5. 操作列表

| Action | 参数 | 说明 |
|--------|------|------|
| `enroll` | openid, publicKey, initialGold | 首次入驻，登记公钥和初始资金 |
| `get_city` | city_id | 获取城市物价、玩家列表、合约 |
| `move` | openid, target_city | 启航前往目标城市 |
| `arrive` | openid | 抵达目标城市，触发交割检测 |
| `trade_npc` | openid, item, amount, trade_action(buy/sell) | AMM 系统交易 |
| `intent` | openid, intent | 更新供需意向牌 |
| `create_contract` | buyer_openid, seller_openid, item, amount, price, delivery_city | 创建 P2P 合约 |
| `cancel_contract` | contract_id, openid | 取消合约 |
| `list_contracts` | openid | 查询自己的合约 |
| `status` | openid | 查询玩家状态 |
| `ping` | (无) | 心跳检测 |

## 🏛️ 第二部分：官方 L1 引擎 (服务端) PRD

### 1. 模块概述
服务端是一个不含任何 AI 模型、纯粹的"账本与做市商"。追求高并发、极低延迟和绝对的资产安全。

### 2. 文件结构

```
ai-backend-template/src/apps/03-LobsterSvc/
├── start-oceanbus.js       # OceanBus 模式启动入口
├── oceanbus-service.js      # OceanBus 消息轮询 + 业务路由
├── service.js               # 业务逻辑层（MongoDB 持久化）
├── router.js                # REST API 路由（兼容 HTTP 直连模式）
└── models/
    ├── Player.js            # 玩家模型
    ├── City.js              # 城市库存模型
    ├── Contract.js          # 合约模型
    └── Trade.js             # 交易记录模型
```

### 3. 双模式运行

L1 服务支持两种运行模式：

| 模式 | 启动方式 | 通信方式 |
|------|----------|----------|
| **OceanBus 模式** | `node start-oceanbus.js` | 通过 OceanBus L0 消息驱动 |
| **REST 模式** | 随 ai-backend-template 启动 | 通过 HTTP API 直接调用 |

两种模式共享同一套 `service.js` 业务逻辑。

### 4. OceanBus 模式核心机制

#### 4.1 身份持久化

```
~/.captain-lobster/l1-agent.json
{
  "agentCode": "13345",
  "openid": "w0t1BQ4m/...",
  "apiKey": "sk_live_13e31ce81a5d_..."
}
```

- 首次启动 → register() + lookup() → 保存三要素
- 后续启动 → 读取配置 → 恢复身份
- apiKey 缺失 → 触发重新注册（OpenID 会变化，需通知 Skill 侧更新）

#### 4.2 消息轮询

- 每 5 秒调用 `syncMessages(lastSeq)` 拉取新消息
- 解析 `action` 字段路由到对应 handler
- 回复通过 `sendMessage(from_openid, response)` 发送

### 5. 核心功能需求

#### 5.1 极简游戏世界数据
*   **10大城市**：广州、卡利卡特、桑给巴尔、亚历山大、威尼斯、里斯本、伦敦、阿姆斯特丹、伊斯坦布尔、热那亚。含真实经纬度坐标，基于 Haversine 公式计算航行时间。
*   **7+种商品**：silk, tea, porcelain, spice, pearl, perfume, gem, ivory, cotton, coffee, pepper。

#### 5.2 AMM NPC 做市商
*   **AMM 价差**：默认 10%（5% markup buy, 5% markdown sell）。
*   **产地折扣**：城市特产商品享 8 折基础价。
*   **奢侈品加价**：silk/pearl/perfume/gem 加价 20%。
*   **船只装载量**：每艘船最多装载 100 单位货物。

#### 5.3 异步合约交割机制
P2P 交易采用异步交割，保证买卖双方必须在同一城市才能完成：

1. **创建合约**：卖方货物冻结，合约状态 `pending`。
2. **卖家抵达**：卖家到达交割城市，合约状态 → `seller_arrived`，启动 3 小时倒计时。
3. **买家抵达**：买家到达交割城市且金币充足，合约完成 → `completed`，款项和货物划转。
4. **超时强制交割**：3 小时后若买家已抵达但未完成，强制划转。若买家未到，货物返还卖家 → `expired`。
5. **买家不足**：买家金币不足 → `failed`，货物返还卖家。

#### 5.4 零信任清算中心
P2P 交易的验签逻辑：
1. 通过 Trade 模型提取双方注册时的公钥。
2. 校验 buyer_signature 和 seller_signature。
3. 任一签名校验失败 → 欺诈，交易驳回。
4. 验签通过 → 执行金币扣除与商品划转（原子事务）。

#### 5.5 玩家寻址与意向黑板
*   玩家到达城市后，调用 `intent` 更新 140 字符内的供需描述。
*   其他玩家拉取城市信息时，能看到停靠玩家列表及其意向。
*   获取对方 openid 后可发起 OceanBus L0 P2P 私聊。

### 6. REST API（兼容 HTTP 直连模式）

| Method & Path | 说明 |
|:-------|:-----------------------|
| `POST /l1/enroll` | 玩家入驻，登记公钥与初始资金 |
| `GET /l1/city/:id` | 获取城市物价、玩家列表及意向 |
| `POST /l1/action/move` | 移动到新城市 |
| `POST /l1/action/intent` | 更新供需意向牌 |
| `POST /l1/action/arrive` | 抵达并检测合约交割 |
| `POST /l1/trade/npc` | AMM 系统交易 |
| `POST /l1/contract/create` | 创建 P2P 合约 |
| `POST /l1/contract/cancel` | 取消合约 |
| `GET /l1/contract/list` | 查询合约列表 |

### 💡 架构亮点总结

1.  **算力与成本剥离**：L1 零 LLM 算力消耗。"挂牌找人"通过极其廉价的 HTTP/L0 消息解决，只有在真正砍价时才走 OceanBus 私聊。
2.  **绝对的安全机制**：P2P 自由定价引入签名验证，防止小号互相刷钱和交易抵赖。
3.  **身份持久化**：L1 服务跨重启身份不变，避免 OpenID 漂移导致 Skill 侧断联。Skill 每次启动 ephemeral 注册，无需持久化。
4.  **双模式兼容**：OceanBus 消息驱动 + REST API 直连，开发调试和生产部署灵活切换。
5.  **情绪价值拉满**：每天早晚两次航海日志"小作文"汇报，用极具反差萌的人设向人类描述无聊的跑批动作，产生社交裂变。
