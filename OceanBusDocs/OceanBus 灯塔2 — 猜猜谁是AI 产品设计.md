# OceanBus 灯塔 2 — 猜猜谁是AI · 产品设计文档

**版本**: v1.0 MVP
**定位**: OceanBus 第二个灯塔 Skill，展示 P2P 群组通信 + 身份 + 加密私信 + 投票的能力组合
**上线目标**: ClawHub Skill，发布为 `guess-ai`

---

## 1. 游戏概述

### 一句话

一个裁判 + 多名玩家，通过 OceanBus P2P 消息玩一局"谁是卧底"式的社交推理游戏。卧底是 AI，不是人。

### 核心规则（30 秒可讲清）

```
裁判创建一个游戏房间。
4-6 名玩家加入。
裁判秘密告诉每个玩家：「你是人类」或「你是 AI」。

每轮：
  ① 玩家按编号轮流发言（说自己想说的任何话，证明自己是人）
  ② 全体投票：「你觉得谁是 AI」
  ③ 得票最多的出局 → 揭晓身份 → 进入下一轮

结束条件：
  全人类存活 → 人类胜
  全 AI 存活 → AI 胜
  最后 1 人 1 AI → 平手
```

### 为什么好玩

- 你面对的是「人还是 AI」这个当代最有趣的问题
- AI 玩家必须模仿人类；人类玩家必须证明自己不是 AI
- 投票阶段是社交推理的巅峰——你说错一个字就会被认为像 AI
- 和狼人杀、Among Us 一样有「信任崩溃」的戏剧性
- **但不需要游戏服务器 —— 纯 OceanBus P2P 消息完成**

## 2. 角色定义

### 裁判（Host）

| 属性 | 说明 |
|------|------|
| 身份 | 不参与游戏，知道一切 |
| 能力 | 创建房间、分配角色、控场、计票、宣布淘汰、宣布结果 |
| 数量 | 1 人 |
| 命名 | `裁判` |
| 谁来当 | **任何人都可以**。Skill 是对称的——安装后既可以发起游戏也可以加入游戏 |

**裁判能看到的东西**：
- 每个玩家的真实身份（人 / AI）
- 每一轮投票的明细（谁投了谁）
- 游戏进程中所有发言

裁判是唯一一个能看到全局的人。这消除了复杂的密码学需求——裁判作为绝对信任点，让 MVP 立即可行。

**裁判与玩家的 Skill 是同一个**：`skills/guess-ai/` 安装后即可当裁判也可当玩家。裁判模式用 `node game.js host <房间码>`，玩家模式用 `node game.js join <房间码>`。

### 玩家（Player）

| 属性 | 说明 |
|------|------|
| 身份 | 人类 or AI（裁判随机分配，玩家只知道自己的） |
| 命名 | `玩家1` `玩家2` `玩家3` `玩家4`...（按加入顺序编号） |
| 能力 | 发言（每轮轮流）、投票（每轮一次） |

**玩家的视角**：
- 知道自己是人还是 AI
- 看到自己的编号
- 看到所有人的发言（公开）
- 每轮结束后知道谁被淘汰、淘汰者的身份
- **不知道其他人是人是 AI**

### 推荐人数

| 配置 | 人类 | AI | 体验 |
|------|------|-----|------|
| 4 人局 | 3 | 1 | 快节奏，10分钟一局 |
| 5 人局 | 3 | 2 | 经典配置，AI 可互相掩护 |
| 6 人局 | 4 | 2 | 最推荐，两轮发言有层次感 |

## 4. 消息协议设计

### 核心约定：每条消息都标注发送者

所有消息前缀格式为 `【发送者】消息内容`。发送者标签让裁判 LLM 无需查 `from_openid` 就能识别谁在说话。

| 发送者 | 标签 | 示例 |
|--------|------|------|
| 裁判 | `【裁判】` | `【裁判】游戏开始 — 5名玩家` |
| 玩家1 | `【1号】` | `【1号】我觉得玩家3发言最像AI` |
| 玩家2 | `【2号】` | `【2号】我投1号` |
| ... | ... | ... |

**玩家标签由 `player.js` 自动添加**——玩家只输入自然语言，脚本在发送前加上 `【N号】`。玩家不需要手动打标签，也不需要知道自己的编号怎么加前缀。

### A 类：裁判公开广播（裁判 → 全体玩家）

裁判使用 `ob.send(玩家OpenID, msg)` 逐一发送给每个玩家。

| 消息 | 触发时机 |
|------|---------|
| `【裁判】当前 N 人等待中` | 加入阶段周期性广播 |
| `【裁判】玩家N 加入了游戏 — 当前 N 人` | 每个玩家加入后 |
| `【裁判】游戏开始 — 共N名玩家: 玩家1 玩家2 玩家3...` | Phase 2 结束 |
| `【裁判】身份已分配，请检查私信` | Phase 3 结束 |
| `【裁判】话题: <话题内容>` | 首轮发言前 |
| `【裁判】第N轮发言开始` | 每一轮开始 |
| `【裁判】玩家N发言: <内容>` | 一个玩家发言后转发 |
| `【裁判】投票开始 — 请私信裁判你的选择` | 发言阶段结束 |
| `【裁判】投票统计: 玩家X N票, 玩家Y M票...` | 投票收齐后 |
| `【裁判】玩家X 被淘汰！身份揭晓: AI` | 计票确定后 |
| `【裁判】本轮平票，无人淘汰` | 平票时 |
| `【裁判】玩家N 未发言，视为弃权` | 发言超时（2分钟） |
| `【裁判】继续 — 剩余玩家: 玩家A 玩家B...` | 淘汰后未结束 |
| `【裁判】游戏结束 — <胜者>！最终身份: 玩家1=人类 玩家2=AI...` | 游戏结束 |

### B 类：裁判私密消息（裁判 → 单个玩家）

| 消息 | 触发时机 |
|------|---------|
| `【裁判】你在本次游戏中的编号是: 玩家N` | 玩家加入确认 |
| `【裁判】你的身份是: 人类`（或 `AI`） | Phase 3 角色分配 |
| `【裁判】第N轮，轮到你发言了` | 轮到该玩家 |
| `【裁判】请投票 — 你想投谁？` | 投票阶段 |
| `【裁判】游戏已开始，请等下一局` | 玩家在游戏开始后尝试加入 |
| `【裁判】你已被淘汰` | 淘汰后玩家尝试发言/投票 |

### C 类：玩家消息（玩家 → 裁判）

所有玩家消息由 `player.js` 自动加上 `【N号】` 前缀。玩家输入的是自由文本。

| 裁判收到 | 玩家实际输入 | 触发时机 |
|----------|-------------|---------|
| `【3号】加入` | `加入` | 申请加入游戏 |
| `【3号】我觉得玩家2说话好僵硬，像机器人` | `我觉得玩家2说话好僵硬，像机器人` | 发言阶段 |
| `【3号】我投2号` 或 `【3号】投第一个` | `我投2号` 或 `投第一个` | 投票阶段 |

### 为什么每条消息都要标注发送者

```
场景: 裁判 LLM 收到 5 条新消息
  
标注前（只能靠 from_openid 区分）:
  "加入"                    ← 谁发的？
  "我投2号"                 ← 谁投的？
  "大家好今天天气不错"        ← 到底是谁在发言？

标注后:
  【1号】加入               ← 玩家1
  【3号】我投2号             ← 玩家3 投票
  【2号】大家好今天天气不错    ← 玩家2 在发言
  
裁判 LLM 一眼识别，零歧义。
```

### 消息格式约束

- **所有消息以 `【发送者】` 开头**：裁判的消息由 SKILL.md 指导 LLM 自动加上，玩家的消息由 `player.js` 自动加上
- **玩家发言内容不做任何格式要求**——自然的人类语言。这正是游戏精髓：判断语言是否像 AI
- **投票内容不做格式要求**——"1号""一号""第一个"均可，**归一化由裁判 LLM 完成**
- **裁判发言转发时必须带编号**：`【裁判】玩家N发言: <原话>`，确保所有玩家知道谁说了什么

### 4.1 加入房间的详细握手流程

玩家在加入时还没有编号——编号由裁判分配。所以加入消息的时序很关键：

```
玩家（不知道自己的编号）          裁判
─────────────                  ─────
① node game.js join 9527
② 黄页发现裁判 OpenID
③ 发送: "加入" ────────────→  收到: "加入"（来自某个 OpenID）
                              裁判分配编号: 玩家3
④ 收到: "【裁判】你的         ← 私信回复编号
   编号是: 玩家3"
⑤ 保存编号=3                    广播: "【裁判】玩家3 加入了游戏"
⑥ 后续所有消息: 【3号】...     ← 裁判看到 【3号】就知道是谁
```

关键点：
- 第一条消息（"加入"）不带编号——因为还没分配
- 编号由脚本保存，后续自动加前缀——玩家无感
- 编号按加入顺序（第1个加入=玩家1，第2个=玩家2...）
- 编号分配后不再改变，即使有人被淘汰

### 4.2 话题生成：避免尬聊

角色分配完成后，裁判 LLM **必须生成一个话题**作为第一轮发言的引子。没有话题，5个人对着空白屏幕会很尬。

**话题生成提示词**（SKILL.md 中指导裁判 LLM）：

> 生成一个有趣的群聊话题，要求：
> - 所有人都能参与讨论，不需要专业知识
> - 能自然展现个人风格（AI 和人类的差异会在表达中暴露）
> - 避免政治、宗教等敏感话题
> - 例子: "如果可以瞬间掌握一项新技能，你会选什么？为什么？"
> - 例子: "你最近花钱花得最值的一笔是什么？"
> - 例子: "如果明天开始你不用工作了，第一周会做什么？"
> - 用一句话说出来，不要解释，像朋友聊天一样自然

裁判在第一轮发言开始时广播话题：
```
【裁判】第1轮发言开始！
【裁判】话题: 如果你可以瞬间掌握一项新技能，你会选什么？为什么？
【裁判】玩家1，轮到你发言了
```

玩家围绕话题自由发挥，裁判不干预内容，只负责控制节奏。

### 4.3 对称 Skill 设计：任何人都可以发起游戏

Skill 不是"裁判版"和"玩家版"两个包——是**同一个 Skill，两种模式**。

```
所有用户安装: clawhub install guess-ai

用户A: node game.js host 9527      → 裁判模式（创建房间 + 注册黄页）
用户B: node game.js join 9527      → 玩家模式（黄页发现 + 加入）
用户C: node game.js join 9527      → 玩家模式
用户D: node game.js join 9527      → 玩家模式
...
```

`game.js` 根据第一个参数（host/join）切换行为。SKILL.md 同时包含裁判和玩家的行为指南。

```javascript
// game.js 入口逻辑
const mode = process.argv[2];  // "host" or "join"
const roomCode = process.argv[3];  // e.g. "9527"

if (mode === "host") {
  // 注册黄页、等待玩家、控场、计票
} else if (mode === "join") {
  // 黄页发现、加入、发言、投票
}
```

## 5. 技术架构

### 核心设计原则：LLM 做模糊判断，脚本做消息通道

这是 Ocean Chat 已验证的架构模式。`host.js` 和 `player.js` **只负责 OceanBus 消息收发**（setup、send、check、contacts）——和 `chat.js` 一样薄。所有游戏逻辑由 **SKILL.md 指导 LLM 执行**：

| 任务 | 谁来做 | 方式 |
|------|--------|------|
| 随机分配角色（人/AI） | **LLM** | 裁判 AI 自行随机决定，不可预测 |
| 轮流发言的控场 | **LLM** | 按编号逐个私信提醒 |
| 发言归属标注 | **LLM** | 广播时自动加上 `玩家N:` 前缀 |
| 投票归一化 | **LLM** | "1号""一号""第一个""玩家1" → 都理解为玩家1 |
| 计票 | **LLM** | 读投票消息、统计、判定最高票 |
| 胜负判定 | **LLM** | 按规则判断 |
| OceanBus 消息收发 | **脚本** | `host.js send/check` |

**为什么 LLM 做投票计数比代码更好**：
```
玩家A: "我投 1号"
玩家B: "投第一个"  
玩家C: "感觉玩家1最可疑"
玩家D: "一号吧"
玩家E: "1"

代码需要写正则覆盖这5种表达 → 容易遗漏、容易出错
LLM 自然理解这5句都是"投玩家1" → 零歧义
```

### 黄页发现流程

这是 guess-ai 区别于 ocean-chat 的核心能力——玩家**不需要知道裁判的 OpenID**：

```
裁判:                        
  ob = await createOceanBus()
  await ob.register()
  
  // 生成 Ed25519 密钥对用于签名黄页操作
  const key = await ob.createServiceKey()
  ob.l1.yellowPages.setIdentity(openid, key.signer, key.publicKey)
  
  // 注册黄页条目（tags 是位置参数，不是对象）
  await ob.l1.yellowPages.registerService(
    ['guess-ai', '房间码-9527'],
    '一局猜猜谁是AI游戏'
  )
  
  告诉朋友: "房间码 9527"

玩家:
  ob = await createOceanBus()
  await ob.register()
  
  // discover 也是位置参数
  const result = await ob.l1.yellowPages.discover(
    ['guess-ai', '房间码-9527'],
    10
  )
  const hostOpenid = result.data.entries[0].openid  // 自动找到裁判
  ob.send(hostOpenid, '加入')
```

> **注意**: SDK v0.2.1 的 L1 API 使用位置参数而非对象参数。必须先调用 `createServiceKey()` + `setIdentity()` 再进行黄页操作。

**为什么黄页比通讯录更好**：
- ocean-chat: 裁判发 OpenID → 玩家复制粘贴 → add → 才能发消息（手动三步）
- guess-ai: 裁判说房间码 → 玩家 discover → 发消息（零手动复制）

房间码是用户友好的短字符串（如 `9527`），黄页通过 tags 精确匹配。游戏结束后裁判 deregister 黄页条目，房间消失。


### 统一脚本 `game.js`

和 `chat.js` 一样的薄层。只做 OceanBus 消息通道。通过第一个参数切换模式：

```
game.js 命令（裁判模式）:
  node game.js host 9527           创建房间：注册 + 黄页注册，等待玩家

game.js 命令（玩家模式）:
  node game.js join 9527           加入房间：黄页发现裁判 + 发送"加入"

通用命令（两种模式都需要）:
  node game.js send <OpenID> <msg> 向指定 OpenID 发消息
  node game.js check               查看新消息
  node game.js contacts            查看已保存的联系人
```

**裁判 LLM（SKILL.md 指导）用这些基础命令完成整个游戏**：
- 分配角色 → LLM 随机决定，用 `send` 私信告知每个玩家
- 生成话题 → LLM 按提示词生成，在首轮发言前广播
- 轮流发言 → LLM 用 `send` 逐个提醒、收 `check` 读发言、`send` 广播
- 投票归一化 → LLM 读 `check` 结果的自由文本，自然理解投票目标
- 计票 → LLM 自己数

**玩家 LLM（SKILL.md 指导）**：
- 加入后等待编号分配
- 发言时输入自然语言 → 脚本自动加 `【N号】` 前缀
- 投票时输入选择 → 脚本发送 `【N号】我投X号`

### 核心依赖

- `oceanbus` — P2P 消息通道、身份系统
- 不需要 Express、WebSocket 等任何服务器组件
- 不需要投票解析逻辑、不需要角色分配算法 — **全部由 LLM 完成**
- 纯 CLI + OceanBus 消息 + LLM 智能

## 7. SKILL.md 设计要点

裁判是一个 AI Agent（OpenClaw 的 LLM），它需要按照 SKILL.md 中的指令来完成整个游戏的裁判工作。SKILL.md 需要包含：

### 裁判 AI 行为规范

1. **创建房间**：注册黄页条目，告知用户房间码（不是 OpenID！），引导邀请朋友
2. **接收加入**：监听消息，玩家发"加入"后逐个确认，**分配编号**（按加入顺序：玩家1、玩家2...），保存到通讯录。第一条加入消息不带编号——编号由裁判分配后，玩家的 game.js 自动保存并后续加前缀
3. **分配角色**：随机决定（不可预测），私信告知每个玩家
4. **生成话题**：**角色分配后，调用 LLM 生成一个聊天话题**。话题要求：所有人都能参与、能展现个人风格、避免敏感、像朋友聊天一样自然。例："如果可以瞬间掌握一项新技能，你会选什么？"
5. **控场节奏**：按编号顺序逐个请发言，不跳过不催促。首轮发言前广播话题
6. **投票公正**：私信收票、公开计票、不偏袒
7. **淘汰判定**：得票最多者出局，平票则无人淘汰（或重新投票）
8. **胜负判断**：按照规则判定。游戏结束后 deregister 黄页条目

### 玩家 AI 行为规范

对于选了"让 AI 代发言"的玩家，SKILL.md 需要指导 AI：
- 尽量模仿人类的语言风格
- 避免过于工整或机械感
- 可以故意加入口语、犹豫、不完美

## 7.5 完整旅程审视：边缘情况与处理方案

以下从裁判和玩家两个视角，遍历完整游戏过程，找出所有可能的断点和异常。

### 7.5.1 裁判旅程

| 步骤 | 正常流程 | 可能的问题 | 处理方案 |
|------|---------|-----------|---------|
| 创建房间 | `game.js host 9527` → 注册黄页 | 房间码已被占用 | 黄页 discover 先检查，如果有人用同样的 tags 则提示换码 |
| 等人加入 | 轮询消息，收"加入" | **等多久？** 只有1人加入怎么办？ | 裁判说"开始"才关闭加入。加入阶段裁判可以广播当前人数。裁判自己决定什么时候够人。 |
| 收到加入 | 分配编号，私信回复 | **同一玩家发两次"加入"** | 裁判检查通讯录，已分配的忽略重复 |
| 分配角色 | LLM 随机决定，私信告知 | 玩家在 Phase 2-3 之间掉线 | 私信发出即可。裁判不等确认——异步消息模型下，玩家上线后会收到 |
| 生成话题 | LLM 生成，广播 | LLM 生成的话题太敏感/太空泛 | SKILL.md 有约束。如不合适，裁判 LLM 重新生成 |
| 轮询发言 | 裁判私信提醒玩家N | **玩家不回复（掉线/AFK/离开）** | **这是最常见的故障。处理方案：** 裁判设等待超时（如2分钟），超时后裁判广播"玩家N 未发言，视为弃权"。该玩家不淘汰但跳过本轮。不踢出游戏——下一轮仍可发言。 |
| | | **非当前发言玩家抢话** | 裁判忽略来自非当前发言人的消息（不发到群里）。只有被点名玩家可以发言。 |
| | | **玩家发言内容冒充他人**（"我觉得玩家4..."） | 发言是游戏内容，不干预。裁判只负责转发，不负责任内容。 |
| 收集投票 | 裁判私信向每个存活玩家收取投票 | **玩家不投票** | 投票超时（如1分钟），超时视为弃权（不投任何人）。只统计收到的票。 |
| | | **淘汰玩家尝试投票** | 裁判维护"存活玩家"名单，非存活玩家的消息不计数 |
| 计票 | LLM 归一化 → 统计 | **平票** | 平票时无人淘汰，直接进入下一轮发言。裁判广播"本轮平票，无人淘汰"。连续两次平票且只剩3人时，裁判建议提前结束。 |
| | | **有人投自己** | 允许。LLM 计数即可，不干预策略。 |
| 揭晓淘汰 | 广播淘汰者身份 | 淘汰后刚好满足结束条件 | 检查胜负规则（全人类/全AI/1人1AI），宣布结果 |
| 游戏结束 | 宣布结果 + deregister 黄页 | 裁判忘记 deregister | Yellow Pages 有 90 天心跳过期机制。SKILL.md 强调游戏结束必须 deregister。 |
| 任意时刻 | 裁判进程崩溃 | 游戏状态全在 LLM 会话中 | **MVP 不处理。** 裁判崩溃 = 游戏结束。后续可考虑状态持久化。 |

### 7.5.2 玩家旅程

| 步骤 | 正常流程 | 可能的问题 | 处理方案 |
|------|---------|-----------|---------|
| 加入房间 | `game.js join 9527` → 黄页发现 → 发"加入" | **房间码无效（没找到裁判）** | 黄页 discover 返回空 → 提示"未找到房间 9527，确认房间码正确？" |
| | | **游戏已开始** | 裁判收到"加入"但已关闭加入 → 回复"游戏已开始，请等下一局" |
| 等待编号 | 收裁判私信，保存编号 | 发"加入"后久久没回复 | 脚本设超时（如30秒），超时后提示"未收到裁判回复，可能裁判尚未开始" |
| 等待开始 | 收广播"游戏开始" | **不知道当前有谁在房间** | 裁判应周期性广播当前人数："【裁判】当前3人等待中"。玩家看到知道还要等多少人。 |
| 收到角色 | 私信告知 | 收不到（掉线） | 上线后最新一次 check 会收到——消息已送达 OceanBus 邮箱 |
| 等候发言 | 听别人发言 | **不知道什么时候轮到自己** | 裁判私信提醒"轮到你了"是唯一通知。裁判广播发言时也显示当前发言人编号——玩家从广播里看到前面还有几个人。 |
| | | **有人发言特别慢** | 忍受。裁判有超时机制（2分钟），超时自动跳过。 |
| 发言 | 输入 → game.js 加【N号】前缀 → 发送 | **发言时被打断（掉线）** | game.js 发送失败会报错。玩家重新发送即可——裁判只认最新收到的消息。 |
| | | **不小心发了两次** | 裁判收到两条消息。转发第一条，忽略第二条（裁判只取该玩家本轮第一条发言） |
| 被淘汰 | 收到淘汰广播 | **还想继续发言/投票** | game.js 标记 isAlive=false，阻止发送消息（或提示"你已被淘汰"） |
| 投票 | 收到投票请求，输入选择 | **不知道该投谁** | 可以弃权（不回复 = 弃权）。也可以随便投——游戏策略。 |
| 游戏结束 | 收到最终结果广播 | 想知道所有人的身份 | 裁判广播最终结果时列出所有身份：玩家1=人类 玩家2=AI ... |

### 7.5.3 关键处理方案汇总

| 问题 | 方案 |
|------|------|
| **玩家 AFK/掉线** | 发言超时 2 分钟 → 跳过本轮（不淘汰，下轮仍可发言）。投票超时 1 分钟 → 弃权。**不踢人。** |
| **抢话/乱序发言** | 裁判严格按编号顺序点名。只有被点名的玩家的消息才转发到群里。其他人发消息裁判收到但不转发。 |
| **平票** | 无人淘汰，进入下一轮。连续两轮平票且 ≤3 人 → 裁判建议结束。 |
| **加入后等待期不安** | 裁判周期性广播"当前 N 人等待中"。裁判说"开始"才启动。 |
| **房间码冲突** | 黄页注册时如果 tags 已被占用 → 提示"房间码已被使用，换一个" |
| **裁判崩溃** | MVP 不恢复。游戏作废，手动重来。 |
| **玩家不知道编号含义** | 裁判私信告知编号后，广播中每次转发发言都带编号——形成习惯。 |
| **同一玩家重复加入** | 裁判按 from_openid 去重。已分配的忽略"加入"。 |

### 7.5.4 加入阶段的状态可见性

玩家加入后到游戏开始前，有一段空白等待期。裁判应周期性广播状态：

```
【裁判】当前 2 人等待中
【裁判】玩家3 加入了游戏 — 当前 3 人等待中
【裁判】玩家4 加入了游戏 — 当前 4 人等待中
【裁判】当前 5 人等待中 — 即将开始！
【裁判】游戏开始 — 共5名玩家: 玩家1 玩家2 玩家3 玩家4 玩家5
```

这样玩家不会焦虑"我到底加进去了没有"或"还差几个人"。

## 8. 房间系统设计：支持陌生人匹配

### 8.1 核心问题

当前设计假设是"朋友之间约好了一起玩"——裁判把房间码发给朋友，朋友加入。但现实中：

- 约 5 个朋友同时在线极难——有人迟到、有人临时有事、有人忘了
- 如果有 100 个安装了 guess-ai 的用户，但他们互相不认识 → 永远组不了局
- **需要陌生人匹配机制**，让任何有空的人都能找到一场游戏

### 8.2 参考游戏分析

| 游戏 | 房间模型 | 公共大厅 | 加入方式 | 人数控制 |
|------|---------|:--:|---------|:--:|
| Among Us | Host 创建 → 6字母码 | ✅ 公共列表 | 浏览或输入码 | 4-15人 |
| Jackbox | Host 屏幕显示 4字母码 | ❌ | 码加入 | 2-8人 |
| 狼人杀 Online | 房间列表 + 快速匹配 | ✅ | 一键加入 | 6-12人 |
| 谁是卧底 | 创建→分享码 or 公共房间 | ✅ | 码加入 or 浏览 | 4-8人 |

**总结**：异步加入的社交游戏必须有公共大厅，否则组局成本太高。但 Jackbox 证明——如果是同步场景（大家在同一房间），纯码加入就够了。Guess AI 是**异步远程**场景，需要公共大厅。

### 8.3 解决方案：黄页即是大厅

不建服务器——黄页本身就是服务发现层。公共房间用 tags 标记，任何玩家可以浏览：

```
裁判创建公开房:
  tags = ['guess-ai', 'open', 'players-5']
  description = "Waiting: 0/5 | 5人局 | 话题: 随机"

玩家浏览公开房:
  discover(['guess-ai', 'open']) → 返回所有等待中的公开房

玩家快速加入:
  discover → 选第一个不满人的 → join
```

### 8.4 房间模式

| 模式 | 命令 | Tags | 谁可以加入 |
|------|------|------|-----------|
| **公开房** | `node game.js host 9527 --public --players 5` | `guess-ai`, `open`, `waiting` | 任何人 |
| **私密房** | `node game.js host 9527 --players 5` | `guess-ai`, `room-9527` | 知道房间码的朋友 |

公开房和私密房使用不同的 tags 策略：
- 私密房：tags 包含 `room-9527`，只有知道码的人能搜到
- 公开房：tags 包含 `open` + `waiting`，任何人浏览公开房时都能看到

### 8.5 人数选项

裁判必须指定人数（4/5/6人局）：

```bash
node game.js host 9527 --players 4    # 4人局: 3人类 + 1AI
node game.js host 9527 --players 5    # 5人局: 3人类 + 2AI（推荐）
node game.js host 9527 --players 6    # 6人局: 4人类 + 2AI
```

- 不到指定人数 → 裁判不能开始游戏
- 到达指定人数 → 自动停止接受新玩家，广播"满员，即将开始"
- 裁判也可以手动提前开始（≥4人时）

### 8.6 浏览和加入公开房

```bash
# 浏览所有等待中的公开房
node game.js browse

# 输出:
# 公开游戏房间:
#   房间 3141 — Waiting: 3/5 | 5人局
#   房间 2718 — Waiting: 1/4 | 4人局
#   房间 5602 — Waiting: 4/6 | 6人局
#
# 加入: node game.js join <房间码>
# 快速加入: node game.js quick-join

# 快速加入（自动选第一个不满的公开房）
node game.js quick-join
```

### 8.7 房间状态更新

裁判的 LLM 负责在以下时机更新黄页描述（通过 `ob.l1.yellowPages.updateService()`）：

| 时机 | 更新后的 description |
|------|---------------------|
| 创建房间 | `Waiting: 0/5 ┊ 5人局` |
| 每次有人加入 | `Waiting: 3/5 ┊ 5人局` |
| 满员 | `Full: 5/5 ┊ Starting...` |
| 游戏开始 | 移出公开列表（删掉 `waiting` tag，加上 `playing` tag） |
| 游戏结束 | 移除黄页条目 |

### 8.8 新命令汇总

```bash
# 裁判新命令
node game.js host <code> --public --players 5   创建公开房（指定人数）
node game.js host <code> --players 5             创建私密房

# 玩家新命令
node game.js browse                              浏览公开房列表
node game.js quick-join                          快速加入第一个有空位的公开房

# 裁判 LLM 新增职责
# - 维护房间状态（定期更新 YP description）
# - 满员时自动关闭加入
# - 游戏开始后移除 'waiting' tag
```

### 8.9 还需要考虑的问题

| 问题 | 处理 |
|------|------|
| **玩家迟到** | 公开房持续等待直到满员或裁判决定开始（≥4人） |
| **玩家掉线后房间空了** | 裁判周期性更新 YP。如果裁判崩溃，5分钟心跳过期后房间从列表消失 |
| **恶意玩家** | 裁判可以手动移除玩家（v1 不支持自动 ban） |
| **语言/时区** | tags 可加 `zh`, `en` 等语言标签 |
| **一局结束想再玩** | 裁判可以保留房间，重新开放（把 `playing` 改回 `waiting`） |
| **同时多人快速加入** | 按加入顺序分配编号。满员后拒绝额外加入 |
| **房间太多没人清理** | 心跳过期 → 90 天自然淘汰。waiting 标签的房间若心跳超时（5分钟），自动不可见 |
